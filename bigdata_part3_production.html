DevOps для Data — Docker · K8s · Terraform

Межа між Data Engineer і Data Platform Engineer у 2026 — саме тут. Без вміння упакувати pipeline у контейнер, деплоїти у Kubernetes і описати інфраструктуру кодом — ви не зможете підтримувати production-ready системи.

Чому Data Engineer має вміти DevOps

П'ять років тому відповідальність DE зупинялась на «написав Spark-джоб». Тепер — це повна операційна відповідальність: продовуєте свої pipelines, їх треба деплоїти, апдейтити, відкочувати при помилці, моніторити SLA, масштабувати під навантаження. Усі Modern Data Stack сервіси (Airflow, Spark, Kafka, Flink, dbt) — контейнеризовані і живуть у Kubernetes. Якщо ви не керуєте цим самі — хтось інший керує, і цей «хтось» повільно, дорого і не розуміє data-специфіки.

Docker — базові принципи для data workloads

Docker ви, ймовірно, вже бачили. Повторимо акценти, важливі саме для data:

  • Multi-stage builds — обов'язково. Base-образ з JDK+Spark компілюється у builder-стадії, потім копіюємо тільки артефакти в slim-образ. Скорочує розмір у 3-5×.
  • Base images для data: python:3.12-slim, bitnami/spark, apache/airflow, confluentinc/cp-kafka. Уникайте :latest — завжди пінуйте версію.
  • Volumes для state — Airflow metadata, Kafka logs, Zookeeper state. Ніколи всередині контейнера.
  • .dockerignore — не тягніть у контекст .venv, __pycache__, .git, data/. Скорочує build-час у 10×.
  • Non-root user — безпека. RUN useradd -r appuser && USER appuser. Інакше K8s може заблокувати з PSP/PSA.
  • Health checks — HEALTHCHECK інструкція або livenessProbe у K8s.

Kubernetes — мінімальна архітектура, яка треба

Kubernetes · control plane + worker nodes для типового data-стеку

Ключові примітиви K8s для data

Po
Pod

Мінімальна одиниця деплою — 1 або кілька контейнерів з shared network/volumes. Рідко створюєш напряму, завжди через Deployment/StatefulSet.

D
Deployment

Stateless — для Airflow-webserver, dbt-runner. Підтримує rolling updates, replicas. Pods — взаємозамінні.

S
StatefulSet

Для Kafka, ZooKeeper, ClickHouse — pods мають стабільні імена (kafka-0, kafka-1) і persistent volumes, які не губляться.

J
Job / CronJob

Для batch Spark jobs і ETL-скриптів. Виконується до завершення, потім видаляється. CronJob — планувальник.

Sv
Service

Мережева абстракція — стабільна IP/DNS для групи pods. ClusterIP (internal), NodePort, LoadBalancer.

In
Ingress

L7 routing ззовні у кластер. Вказує, як потрапити на Airflow UI, Grafana, DataHub. NGINX Ingress — дефолт.

CM
ConfigMap / Secret

Конфіги і креденшіали окремо від образів. Sealed Secrets або External Secrets Operator для шифрування у Git.

PV
PVC / StorageClass

Persistent storage для stateful workloads. На хмарах мапиться у EBS/PD/Managed Disk.

Spark on Kubernetes

З 2019 року (Spark 2.3+) Kubernetes — перший-класний cluster manager для Spark, поряд з YARN. Рекомендований підхід у 2026 для нових деплойментів.

  • Driver — окремий pod, executor-и — окремі pods. Spark сам робить запит до K8s API при старті джобу.
  • Для керування через Airflow — SparkKubernetesOperator або Spark Operator.
  • Переваги над YARN: не треба окремий кластер, краща інтеграція з sidecar-контейнерами (Envoy, Jaeger), native RBAC.
  • Dynamic allocation — executor-и створюються/вбиваються за потребою.

Helm — пакетний менеджер для K8s

YAML-маніфести K8s швидко стають некерованими для складних стеків. Helm charts — шаблонизовані YAML + values для різних environments (dev/staging/prod). Де-факто стандарт для деплою open-source сервісів.

  • Airflow: official helm chart — повний стек за 1 команду.
  • Kafka: Strimzi Operator — стандарт у 2026. Керує Kafka, ZooKeeper/KRaft, Schema Registry, Kafka Connect.
  • Spark: Spark Operator.
  • Flink: Flink Kubernetes Operator.
  • ClickHouse: Altinity Operator.
  • DataHub: acryldata/datahub chart.
  • Prometheus + Grafana: kube-prometheus-stack — все в одному.

Terraform — Infrastructure as Code

Один із ключових навичок сучасного DE. Terraform (HashiCorp) описує будь-яку інфраструктуру (AWS, GCP, Azure, K8s, Snowflake, Databricks) у декларативних HCL-файлах. Ви запушили зміни в Git → pipeline підняв інфраструктуру.

# S3 bucket для ShopWave data lake
resource "aws_s3_bucket" "shopwave_lake" {
  bucket = "shopwave-${var.env}-lake"
  tags   = {
    Project = "shopwave"
    Env     = var.env
    Owner   = "data-team"
  }
}

resource "aws_s3_bucket_versioning" "shopwave_lake" {
  bucket = aws_s3_bucket.shopwave_lake.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_lifecycle_configuration" "shopwave_lake" {
  bucket = aws_s3_bucket.shopwave_lake.id
  rule {
    id     = "move-old-to-glacier"
    status = "Enabled"
    transition {
      days          = 90
      storage_class = "GLACIER"
    }
  }
}
Основні концепції Terraform
  • Resources — що створити (VPC, S3 bucket, EKS cluster, IAM role).
  • State (terraform.tfstate) — поточний стан. ОБОВ'ЯЗКОВО тримати у remote backend (S3 + DynamoDB lock), не у Git.
  • Modules — переюзабельні пакети ресурсів. terraform-aws-modules/vpc/aws, eks/aws — написані AWS-спільнотою.
  • Variables + outputs — параметризація.
  • Workspaces або (краще) окремі backends для dev/staging/prod.
⚠ Warning

Ніколи не комітьте terraform.tfstate у Git — там у plain text лежать паролі, access keys, endpoints. Використовуйте S3 backend з encrypt=true + DynamoDB state lock. Це правило №1 у Terraform.

Альтернативи Terraform
  • Pulumi — те саме, але справжніми мовами (Python, TypeScript, Go). Хороший вибір для дев-команд, які не хочуть вчити HCL.
  • AWS CDK — AWS-only, але глибока інтеграція.
  • Crossplane — IaC як Kubernetes CRDs. Інфраструктура керується тим самим API, що й pods.
  • OpenTofu — форк Terraform після ліцензійних змін HashiCorp 2023. Спільнотний, open-source. У 2026 багато команд мігрують з TF на OpenTofu.

CI/CD для data pipelines

Коли у вас є Terraform + Kubernetes + Helm + Python-код — ручний деплой неможливий. Типова CI/CD-схема для data team:

  1. Push у feature branch → GitHub Actions: lint (ruff), type-check (mypy), unit tests (pytest)
  2. Pre-commit hooks локально: black, isort, terraform fmt, sqlfluff
  3. PR → integration tests на ephemeral environment (preview K8s namespace)
  4. Merge у main → побудова Docker-образів, пуш у ECR/GCR/ACR з git SHA як тегом
  5. Автоматичний deploy у staging через ArgoCD (GitOps) або kubectl apply
  6. Smoke tests на staging → manual approval → deploy у prod
  7. Spark-джоби і dbt-моделі — тести на sample даних перед деплоєм

Data testing — Great Expectations і dbt tests

Код тестуєте через pytest, але дані треба тестувати окремо. На проді трапляється: джерело раптом почало надсилати null замість email, схема змінилась на один байт, дублі пролізли. Без data-testing — ви дізнаєтесь від бізнесу через тиждень, коли звіти покажуть дивне.

  • Great Expectations — бібліотека, де декларативно описуєте expectations про дані (expect_column_values_to_not_be_null, expect_values_in_set, expect_row_count_to_be_between). Генерує Data Docs HTML-репорт.
  • dbt tests — для dbt-моделей: unique, not_null, accepted_values, relationships. Custom tests через SQL.
  • Soda Core — YAML-based альтернатива, легковаговий.
  • Elementary — спостережливість поверх dbt-tests.
  • Тригер: якщо тест fail, Airflow зупиняє downstream tasks і шле alert у Slack.

Моніторинг: Prometheus + Grafana + Alertmanager

Стандартна тріада SRE-світу. Для data-платформ треба моніторити три групи метрик:

I
Infrastructure

CPU, RAM, disk, network pods. node_exporter, kube-state-metrics.

A
Application

Spark job duration, Kafka lag, Flink checkpoint size, Airflow DAG success rate. StatsD exporter, Micrometer.

B
Business / Data

Row count /день, GMV delta, null-rate. Прописуєте запити у dashboards + alerts на аномалії.

Альтернатива для cloud-native — DataDog (зручно, але дорого) або Grafana Cloud. Для pure data observability — Monte Carlo, Metaplane, Bigeye, open-source OpenLineage.

K8s Troubleshooting — топ-10 проблем для data workloads

Перша реакція на будь-яку K8s проблему: kubectl describe pod <pod-name> → kubectl logs <pod-name> --previous. 90% відповідей там.
OOMKilled — Spark Executor
Pod вбитий через перевищення memory limit. Діагностика: kubectl describe pod <pod> → Events → "OOMKilled". Фікс: збільшити spark.executor.memory і відповідно resources.limits.memory у SparkApplication CRD. Правило: limits.memory = executor.memory + executor.memoryOverhead (мінімум 10%). Наприклад: executor.memory=4g → limit=5g.
CrashLoopBackOff — будь-який Pod
Pod стартує і падає, K8s перезапускає з exponential backoff. Діагностика: kubectl logs <pod> --previous — логи останнього контейнера до краша. Типові причини: bad env variable, missing secret, wrong entrypoint, Python exception в init-коді. Фікс залежить від логу.
Pending Pod — не можна запланувати
kubectl describe pod <pod> → Events → "0/5 nodes are available: 3 Insufficient cpu, 2 node(s) had taint". Причини: 1) resource requests перевищують доступне (збільшити node pool або зменшити requests), 2) taint на нодах без відповідного toleration (додати toleration у spec), 3) nodeSelector або affinity не відповідає (перевірити node labels: kubectl get nodes --show-labels).
ImagePullBackOff — не може завантажити image
Pod не може pull образ. Причини: 1) image не існує або wrong tag → перевірити registry, 2) private registry без credentials → додати imagePullSecret: kubectl create secret docker-registry ecr-secret --docker-server=<ecr-url> ... + spec.imagePullSecrets, 3) ECR image expired (12h TTL на external accounts) → оновити auth token через AWS CLI.
Kafka StatefulSet PVC не монтується
Kafka broker pod Pending або не стартує. kubectl describe pod kafka-0 → "PVC not found" або "can't be bound". Перевірити: kubectl get pvc → статус. Якщо Pending: StorageClass не знаходить PV → kubectl get sc, переконатись що provisioner встановлений (EBS CSI driver для AWS). Якщо volume у wrong AZ — видалити PVC і перестворити StatefulSet у правильній AZ.
Airflow Worker Pod зависає — task Stuck
DAG task показує "running" 12+ годин. kubectl get pods -n airflow → знайти worker pod → kubectl logs <worker-pod>. Якщо logs порожній → worker загинув (OOM або eviction) а Airflow не отримав callback. Фікс: перевірити pod events на Eviction, збільшити worker memory limits, налаштувати executor_task_timeout щоб Airflow виявляв "orphaned" tasks.
Spark Executor Lost — Spot Instance Preemption
Spark executor lost через AWS Spot interruption (2-min warning). Spark не знає чому executor зник — просто retries task. Фікс: spark.task.maxFailures=4 (default 4 retries), spark.dynamicAllocation.enabled=true (автозамінити executor), використовуйте mix On-Demand (driver) + Spot (executors), налаштуйте Node Termination Handler для graceful draining.
Service not accessible / DNS не резолвиться
kubectl exec -it <pod> -- curl http://my-service:8080 → timeout. Перевірити: 1) Service існує kubectl get svc, 2) selector відповідає pod labels kubectl describe svc my-service, 3) Pod endpoints kubectl get endpoints my-service — якщо порожній, selector не матчить, 4) NetworkPolicy не блокує трафік kubectl get networkpolicies.
Init Container не завершується
Pod застряг у Init:0/1. Init containers перевіряють prerequisites (БД ready, secret існує). kubectl logs <pod> -c init-wait-db. Типові проблеми: БД ще не запустилась (треба чекати), wrong database URL у env, TCP порт заблокований NetworkPolicy.
Horisontal Pod Autoscaler не масштабує
kubectl describe hpa <hpa-name> → перевірити Events. Причини: 1) metrics-server не встановлений → kubectl top pods не працює, 2) Pods не мають resource requests → HPA не може обчислити % CPU, 3) minReplicas = maxReplicas (забули різниці). Для custom metrics (Kafka lag → Spark workers) — потрібен KEDA замість HPA.

GitHub Actions CI/CD — повний пайплайн

# .github/workflows/data-pipeline-ci.yml
name: Data Pipeline CI/CD

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  AWS_REGION: eu-west-1
  ECR_REGISTRY: 123456789.dkr.ecr.eu-west-1.amazonaws.com
  IMAGE_NAME: shopwave-spark-jobs

jobs:
  # ─── Lint & Test ─────────────────────────────────────
  lint-and-test:
    name: Lint + Unit Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Python 3.11
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: pip

      - name: Install dependencies
        run: pip install -r requirements-dev.txt

      - name: Ruff lint
        run: ruff check ./shopwave --output-format=github

      - name: mypy type check
        run: mypy ./shopwave --ignore-missing-imports

      - name: pytest unit tests
        run: pytest tests/unit -v --tb=short --cov=shopwave --cov-report=xml

      - name: dbt compile (validate SQL)
        run: |
          cd dbt_project
          dbt deps
          dbt compile --profiles-dir ./profiles --target ci

  # ─── Build & Push Docker ──────────────────────────────
  build-and-push:
    name: Build Docker → ECR
    runs-on: ubuntu-latest
    needs: lint-and-test
    if: github.ref == 'refs/heads/main'
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id:     ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Login to Amazon ECR
        id: login-ecr
        uses: aws-actions/amazon-ecr-login@v2

      - name: Build and push Docker image
        id: meta
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.ECR_REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
            ${{ env.ECR_REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ─── Integration Tests on Staging ────────────────────
  integration-tests:
    name: Integration Tests (staging)
    runs-on: ubuntu-latest
    needs: build-and-push
    environment: staging

    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS + kubectl
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id:     ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: ${{ env.AWS_REGION }}

      - name: Update kubeconfig (EKS staging)
        run: aws eks update-kubeconfig --name shopwave-staging --region $AWS_REGION

      - name: Run integration tests
        run: pytest tests/integration -v --timeout=300
        env:
          SPARK_MASTER: k8s://https://staging-k8s-api:443
          IMAGE_TAG: ${{ github.sha }}

  # ─── Deploy via ArgoCD ────────────────────────────────
  deploy-production:
    name: Deploy → Production (ArgoCD)
    runs-on: ubuntu-latest
    needs: integration-tests
    environment:
      name: production
      url: https://argocd.shopwave.internal

    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GH_PAT }}

🎯 Interview Q&A — Модуль 9: DevOps для Data
Як K8s вирішує resource management для Spark jobs?
Spark on Kubernetes: Driver Pod + динамічно створені Executor Pods. Кожен Pod має resource requests (гарантовані) і limits (максимум). Spark configs: spark.driver.memory, spark.executor.memory, spark.driver.cores, spark.executor.cores. K8s scheduler розміщує pods на нодах де є вільні ресурси. Spark Operator (Kubeflow) спрощує управління SparkApplication CRD. Dynamic Allocation: spark.dynamicAllocation.enabled=true → Executor pods автоматично з'являються під навантаженням і видаляються після idle. Переваги K8s vs YARN: ізоляція, multi-tenancy, легша конфігурація, використання тих самих нод для різних workloads (Spark + Airflow workers + Kafka).
GitOps та ArgoCD — як це організовано?
GitOps: Git репозиторій є єдиним джерелом правди для стану infrastructure та deployments. Ніяких ручних kubectl apply у prod — тільки через PR. ArgoCD: controller у K8s cluster, постійно порівнює desired state (Git) з actual state (cluster). При divergence — alerting або auto-sync. Workflow: розробник змінює Helm values у Git → PR → review → merge → ArgoCD виявляє зміну → синхронізує cluster. Переваги: аудит через git history, rollback = git revert, preview environments через Argo ApplicationSet, automated sync для staging/manual для prod. ApplicationSet дозволяє один шаблон для кількох environments (dev/staging/prod).
Terraform state — що це і чому remote backend обов'язковий?
terraform.tfstate — JSON файл, що описує реальний стан infrastructure (ресурси, їхні ID, залежності). Terraform порівнює desired (код) з actual (state) і обчислює plan. Проблеми з local state: 1) Якщо два інженери запускають terraform apply одночасно — конфлікт і corrupted state, 2) Якщо ноутбук загорівся — state втрачено і треба import всіх ресурсів вручну. Remote backend (S3 + DynamoDB): state зберігається у S3 (versioned bucket), DynamoDB state lock не дає двом apply одночасно, state encrypted at rest (encrypt=true). Ніколи не комітьте tfstate у Git — там credentials у plain text.
Як організувати CI/CD для data pipelines?
Типовий pipeline: 1) Lint + unit tests (ruff, pytest) — швидко, <2 хв, блокує PR при fail. 2) dbt compile — перевірити що SQL компілюється. 3) Build + push Docker image з git SHA як тегом. 4) Deploy в staging + integration tests — Spark job на sample даних, dbt run на staging DWH. 5) Manual approval (environment protection) → deploy in prod через ArgoCD. Дані-специфічне: тести на sample data (<1% prod), data quality checks після deploy, schema compatibility tests (Avro schema registry), DAG validation (Airflow dag-test). Секрети — тільки через GitHub Secrets або AWS Secrets Manager, ніколи у коді.
Stateful vs Stateless workloads у K8s — як вибирати тип ресурсу?
Deployment: stateless workloads — Airflow webserver, API server, Spark driver (тимчасово). Може бути replicated, pod убитий і перестворений без проблем, всі поди ідентичні. StatefulSet: stateful workloads — Kafka brokers, Zookeeper, PostgreSQL, ClickHouse. Гарантує стабільні мережеві ідентифікатори (kafka-0, kafka-1, kafka-2), Persistent Volume Claim пов'язаний з конкретним pod-ом, pods запускаються і зупиняються у порядку. Job/CronJob: Spark jobs, dbt runs, Airflow K8sExecutor tasks — одноразові або за розкладом. Pod стартує, виконує задачу, завершується. Helm charts для data workloads: bitnami/kafka, spark-operator, apache/airflow.
Як моніторити data pipeline у production?
Три рівні: 1) Infrastructure: Prometheus + node_exporter + kube-state-metrics → Grafana. CPU, RAM, disk, network. Alert: pod OOMKilled, node disk >85%, PV near full. 2) Application: Kafka Exporter (lag, ISR), Spark metrics через StatsD → Prometheus, Airflow StatsD, Flink JMX. Alert: consumer lag > threshold, Spark job duration anomaly, DAG SLA miss. 3) Data quality: dbt tests після кожного run, Great Expectations на source data, Monte Carlo для ML-based anomaly detection. Alert: row count drop >20%, null rate spike, business metric anomaly (GMV drop). Playbook: alert → auto-link до runbook → on-call action. PagerDuty для SEV-1, Slack для degradations.
Blue/Green vs Canary deployments для data pipelines — коли що?
Blue/Green: два ідентичних environments (blue = current, green = new). Traffic миттєво перемикається. Rollback = перемикнути назад. Проблема для data pipelines: подвійне споживання Kafka topics або подвійні writes у DWH при перемиканні. Вирішення: consumer groups версіонуються (v1, v2), переключаємо offset після валідації. Canary: поступово збільшуємо % трафіку на нову версію (5% → 25% → 100%). Для streaming: частина partitions читається новим consumer. Дивимося на error rate і lag. Для batch jobs: запускаємо нову версію паралельно на 1-днях даних, порівнюємо вихідні. Зазвичай для data pipelines простіший Blue/Green з feature flags ніж Canary.

Vendor lock-in — риск, який треба усвідомлювати

Кожен сервіс, який ви обираєте, має свій рівень lock-in. Ієрархія:

  • Майже без lock-in: open-source у managed-обгортці (EKS/GKE/AKS, MSK, Elastic, PostgreSQL, Spark через EMR/Dataproc) — мігрувати складно, але реально.
  • Помірний lock-in: S3/GCS/Blob на рівні API — усі підтримують S3-сумісний протокол, але dialect різниться в деталях.
  • Високий lock-in: BigQuery, Redshift, Synapse, Snowflake — свій SQL dialect, свої функції, своя платіжна модель. Міграція = переписати всі запити і ETL.
  • Максимальний lock-in: AWS Step Functions, Azure Data Factory pipelines, IAM-політики — це proprietary формати, які ніде більше не використовуються.

Multi-cloud strategy — корисно чи навіть не пробувати

Multi-cloud має сенс коли...
  • Регуляторні вимоги (частина даних у певній юрисдикції)
  • Disaster recovery (AWS prod + GCP DR — різні failure domains)
  • Вендорські переговори (важіль на знижки)
  • Використовуєте сильні сторони (GCP для BigQuery, AWS для Glacier)
  • Acquisitions / M&A — злили дві компанії з різними стеками
Multi-cloud ускладнює...
  • Egress costs — дані між AWS і GCP $0.09/GB, знищує бюджет
  • Operational overhead — 2 набори IAM, 2 моніторинги, 2 команди експертизи
  • Security surface зростає вдвічі
  • Latency для cross-cloud запитів
  • Team cognitive load

FinOps для data teams

Data-воркоди — головна стаття витрат хмари для багатьох компаній. Базові практики:

  • Tagging everything. Обов'язкові tags: Project, Environment, Owner, CostCenter. Без тегів Cost Explorer марний.
  • Reserved / Committed use — знижка 40-70% за 1-3 роки. Для стабільних workloads обов'язково.
  • Spot / Preemptible — знижка до 80% для fault-tolerant workloads. EMR, Dataproc — ідеально.
  • Auto-scaling і auto-suspend. Snowflake warehouse без auto_suspend=60 = двозначні $ в день даремно.
  • Data lifecycle — S3 lifecycle rules для переведення старого у Glacier. GCS Autoclass.
  • Query optimization — у BigQuery з on-demand pricing один кривий SELECT * на великій таблиці = рахунок на $50.
  • Budget alerts — Cloud Billing alerts на 50%, 80%, 100% бюджету. Автоматичне вимкнення non-prod environments на вихідні.
  • Cost attribution — щомісячні cost-review з усіма командами.

🎯 Interview Q&A — Модуль 10: Хмари
AWS vs GCP vs Azure для Big Data — як обирати?
AWS: найбільша екосистема, найбільший ринок праці. S3 + Glue + EMR/Spark + Kinesis + Redshift — зрілий стек. Мінус: складне ціноутворення, менший ML-фокус. GCP: BigQuery (найкращий serverless DWH), Dataflow (Apache Beam managed), Vertex AI (ML platform), Pub/Sub. Краще для ML-команд та аналітиків. Azure: обов'язковий якщо компанія на Microsoft (Office 365, Azure AD). Synapse Analytics, Azure Data Factory. Корпоративний enterprise. Для навчання: AWS free tier найширший. Для продакшн: де вже є існуюча cloud infrastructure. Тренд 2026: multi-cloud з Iceberg як neutral format — дані в S3, compute вибираємо.
Що таке Reserved Instances/Committed Use та коли це виправдано?
Reserved Instances (AWS) / Committed Use Discounts (GCP) / Reserved Capacity (Azure): передплата на 1 або 3 роки в обмін на знижку 30-72%. On-demand: максимальна гнучкість, платиш за фактичне використання (no commitment). Spot/Preemptible: до 90% дешевше, але може бути відключено. Стратегія: baseline workloads (Kafka brokers, постійні DB, Airflow workers) → Reserved (передбачувано, завжди running). Burst/batch workloads (Spark jobs, ML training) → Spot (дешево, OK якщо перерветься). Dev/staging → On-demand або Spot. Коли виправдано RI: ресурс running >50% часу місяця. Break-even зазвичай ~8-10 місяців. 3-year RI для core infrastructure = швидке payback.
Що таке egress costs і як їх мінімізувати?
Egress = трафік що виходить з хмари назовні або між регіонами. AWS: $0.09/GB для перших 10TB/місяць. Читати дані з S3 на EC2 у тому ж регіоні — безкоштовно. Читати з S3 у різних регіонах — платно ($0.02/GB). Передавати дані назовні (до on-premise або аналітичному інструменту) — $0.09/GB. Мінімізація: 1) Тримати compute у тому ж регіоні що й storage, 2) Cloudflare R2 — S3-сумісне, zero egress fees (підходить для datapipe до CDN), 3) Compression перед transfer, 4) VPC endpoints для AWS services (трафік всередині AWS), 5) AWS Direct Connect / GCP Dedicated Interconnect для on-premise — фіксована ціна замість per-GB. Egress costs можуть бути 20-40% cloud bill при неправильній архітектурі.
Managed Kubernetes (EKS/GKE/AKS) vs self-managed — що обирати?
Self-managed K8s: повний контроль, але ваша команда відповідає за control plane HA, updates, etcd backups, certificate rotation. Для 99.9% компаній це зайва робота. Managed (EKS/GKE/AKS): cloud provider керує control plane, HA з коробки, managed upgrades. GKE: найкращий K8s experience, Autopilot mode (serverless K8s — тільки за pod resources, не node), найшвидший час до production. EKS: AWS ecosystem, tight integration з IAM, Load Balancers, EBS CSI. AKS: Azure AD integration, Windows node pools. Рекомендація 2026: GKE для нових проектів (Autopilot + GKE Standard), EKS якщо вже у AWS-стеку. Ціна managed K8s control plane: GKE безкоштовно, EKS $0.10/год ($73/місяць), AKS безкоштовно.
FinOps — як контролювати cloud costs у data team?
Типові проблеми: forgot to stop dev cluster → $5k за вихідні, DWH warehouse без auto-suspend → $2k/місяць idle, Spark job reads entire table без partition pruning → $100 замість $1. Практики: 1) Tagging — кожен ресурс tagged: team, project, environment, cost-center. Cloud billing за тегами = видно хто скільки витрачає. 2) Budget alerts — 50/80/100% → Slack alert → email C-level при 100%. 3) Auto-shutdown non-prod: Lambda/Cloud Function вимикає dev clusters о 20:00 і вихідні. 4) Cost-aware code: BigQuery → SELECT конкретні колонки (не *), LIMIT в exploration, тільки потрібні партиції. 5) Rightsizing — AWS Compute Optimizer аналізує CPU/RAM usage і рекомендує зменшити. 6) Waste review щомісяця: unused EBS volumes, orphaned Elastic IPs, idle DWH warehouses.

Data Governance & Security

Побудувати платформу — пів справи. У 2026 під GDPR, CCPA, NIS2 і секторальні регуляції (PCI DSS, HIPAA) — треба ще знати хто, коли і навіщо читав які дані, і вміти швидко видалити конкретного користувача на його запит. Без цього production — юридичний ризик.

Що таке Data Governance і чому це не "paperwork"

Data governance — це система відповідальності за дані у компанії: хто власник, хто може змінювати схему, хто має доступ, як ми відстежуємо походження. У 2024-2026 це перестало бути enterprise-онлі: навіть стартапи з 20 людьми потребують базового governance, тому що один leak email-ів = штраф до 4% річного обороту.

GDPR, CCPA, NIS2 — практичні вимоги

R
Right to Access

Користувач має право отримати усі свої дані у структурованому вигляді за 30 днів. Потрібен pipeline, який збирає дані по всіх таблицях за user_id.

F
Right to be Forgotten

Користувач може вимагати видалення. У lake-архітектурі це справжній виклик — треба фізично прати з Parquet-файлів. Iceberg DELETE + expire_snapshots + VACUUM.

C
Consent tracking

Для кожного user треба знати, на що він дав згоду (marketing, analytics, third-party). Зберігати з timestamp, мати API для відзиву.

D
Data retention

Особисті дані — не навічно. GDPR каже «не довше ніж потрібно». Треба політика: orders 7 років (fiscal), events 13 місяців, session-cookies 90 днів.

N
Notification

Breach notification — 72 години. Потрібен audit-лог ВСІХ читання чутливих таблиць, щоб швидко зрозуміти масштаб.

DP
DPIA

Data Protection Impact Assessment — перед запуском нового feature з PII треба оцінити риски. Документ зазвичай пише Legal+DPO разом з DE.

⚠ Важливо — ідеальне не буває

Ця секція — оглядова. Не беріть на себе роль юриста. У серйозних проектах завжди є DPO (Data Protection Officer) або юридичний відділ. Ваша задача як DE — знати достатньо, щоб правильно задавати запитання і знати, що технічно можливо.

PII detection та маскування

Personally Identifiable Information — імена, email, телефони, адреси, IP, cookie-ідентифікатори, геодані. Політика має бути: PII не попадає у зонах без явного дозволу і завжди маскується на шляхах до BI/ML.

Техніки маскування
  • Hashing: SHA256(email + salt). Детермінований — один email завжди дає той самий хеш, можна joins. Необортний.
  • Tokenization: замінити email на token, а мапінг тримати у окремому захищеному vault.
  • Redaction: a***@gmail.com. Читабельно, але не joinable.
  • Format-preserving encryption (FPE): зашифроване має той самий формат (555-12-3456 залишається у форматі SSN). Для схем, які перевіряють формат.
  • Differential privacy: додаєш шум до агрегатів, щоб окремого користувача не можна було реідентифікувати.
  • Synthetic data: згенеровані дані з тією самою статистикою — для dev/test-середовищ.
Автоматична детекція PII
  • AWS Macie, GCP Cloud DLP, Azure Purview — managed ML-сервіси, які сканують сховища і знаходять PII у неструктурованих даних.
  • Presidio (Microsoft) — open-source alternative, працює локально.
  • DataHub має інтеграцію з PII-детекцією і додає теги до колонок.
  • Наслідок: автоматично позначені колонки йдуть через masking-layer у DWH-view перед BI.

IAM · RBAC · ABAC

Три рівня контролю доступу:

IAM
IAM (Identity)

Хто ти? Users, groups, service accounts. SSO через SAML/OIDC (Okta, Auth0, Google). MFA всюди.

R
RBAC (Role-Based)

Що ти можеш робити за роллю. analyst читає gold, engineer пише у silver, admin може все. Стандарт для більшості випадків.

A
ABAC (Attribute-Based)

Більш гнучко: доступ залежить від атрибутів (country=UA, department=marketing, time < 18:00). Для сильно регульованих даних або мультитенантності.

Шифрування — at rest, in transit, at use

  • At rest — дефолт у всіх managed-сервісах (S3, BigQuery, Snowflake). Використовуйте customer-managed keys (CMK) у KMS для справжнього контролю: якщо ключ видалено → дані недоступні навіть провайдеру.
  • In transit — TLS 1.2+ всюди. Внутрішньокластерний трафік у K8s — через service mesh (Istio, Linkerd) з mTLS.
  • Column-level encryption — для sensitive колонок (SSN, card numbers). Snowflake має external tokenization, BigQuery — AEAD-функції.
  • At use — confidential computing (Intel SGX, AMD SEV). Дані розшифровані тільки всередині enclave. Дорого, нішево.

Audit logs та compliance evidence

  • Всі запити у DWH, доступи до lake, зміни у IAM — логуються у централізований audit trail.
  • AWS CloudTrail, GCP Audit Logs, Azure Activity Logs — native-рішення.
  • Snowflake, BigQuery мають свої access history.
  • Logs йдуть у окремий immutable bucket з object lock — щоб навіть admin не міг замести сліди.
  • SIEM (Splunk, Datadog, Elastic) — для real-time-алертів на підозрілі патерни.

Data Mesh — організаційний погляд

Концепція Zhamak Dehghani (2019). Замість одного централізованого data-team з single data platform — data як продукт, який власне data-domain team випускає для інших. Наприклад, checkout-team випускає orders-dataset як офіційний продукт з SLA, schema contract, документацією.

  • Не технологія, а підхід до організації роботи. Можна реалізувати поверх будь-якого стеку.
  • Працює в великих компаніях (500+), маленьким командам — оверкіл.
  • Федеративний governance замість централізованого.
  • Найуспішніші приклади: Zalando, Netflix, HelloFresh.

# ── Batch scanning Parquet-файлів у S3 ────────────────────────────
def scan_parquet_for_pii(s3_path: str) -> dict:
    df = pd.read_parquet(s3_path)
    pii_columns = {}

    for col in df.select_dtypes(include="object").columns:
        sample = df[col].dropna().head(100).astype(str).tolist()
        col_pii_types = set()
        for text in sample:
            results = analyzer.analyze(text=text, language="en")
            for r in results:
                if r.score > 0.7:
                    col_pii_types.add(r.entity_type)
        if col_pii_types:
            pii_columns[col] = list(col_pii_types)

    return pii_columns

🎯 Interview Q&A — Модуль 11: Data Governance & Security
Як реалізувати RBAC у data platform?
RBAC (Role-Based Access Control) для data: 1) Cloud IAM: AWS IAM roles/policies, GCP IAM, Azure RBAC — доступ до S3 buckets, GCS, BigQuery datasets на рівні cloud. Принцип least privilege: сервіс-акаунт Spark-джобу має тільки read на bronze S3 + write на silver. 2) Database RBAC: Snowflake GRANT, ClickHouse RBAC, BigQuery dataset permissions. Наприклад: role analyst → SELECT on gold schema only. role engineer → SELECT/INSERT on silver+gold. role admin → all. 3) Data catalog RBAC: DataHub policies — хто може редагувати метадані яких domain. Практика: ролі в Terraform, documentation у DataHub, quarterly access review (кожен квартал перевіряти — чи нема зайвих прав).
GDPR для data engineer — що потрібно знати?
GDPR (General Data Protection Regulation) — закон ЄС про захист персональних даних. Для data engineer критично: 1) Right to erasure (right-to-be-forgotten): отримали запит → повинні видалити user_id з усіх систем: DWH, lake (Iceberg DELETE + VACUUM), Kafka (tombstone message), MongoDB, backups (або задокументувати що backup буде видалено при наступній ротації). 2) Data minimization: збирати тільки потрібні PII. Не тримати email якщо достатньо user_id. 3) PII classification: знати які поля є PII (email, IP, phone, device_id). Presidio для автоматичного detection. 4) Retention policies: не зберігати PII довше ніж потрібно. S3 lifecycle → автоматичне видалення. 5) Encryption at rest + in transit. 6) Audit log: хто, коли, що переглядав/змінював.
Що таке column-level masking і коли використовувати?
Column masking: автоматично замінювати PII значення на маскований вираз залежно від ролі користувача. Приклад: analyst бачить email як "j***@gmail.com", engineer → hash("email"), admin → оригінал. Snowflake: Dynamic Data Masking policies — CREATE MASKING POLICY mask_email AS (val STRING) RETURNS STRING → CASE WHEN current_role() IN ('ADMIN') THEN val WHEN current_role() = 'ENGINEER' THEN SHA256(val) ELSE REGEXP_REPLACE(val, '.@', '***@') END. BigQuery: Column-level security через Policy Tags + Data Catalog taxonomy. ClickHouse: obfuscation functions або column-level GRANT. Коли потрібно: аналітики мають доступ до gold-шару де є PII — дати доступ але не показувати реальні дані. Кращий підхід: PII у bronze тільки, gold де можливо — anonymized.
Як захистити Kafka від несанкціонованого доступу?
Kafka security шари: 1) TLS/SSL — шифрування трафіку між clients і brokers (listeners: SSL://). 2) SASL Authentication — хто має право з'єднуватись: SASL_PLAINTEXT (dev), SASL_SSL (prod). Механізми: SASL/PLAIN (user/password), SASL/SCRAM (bcrypt-hashed), SASL/OAUTHBEARER (JWT, best practice 2026). 3) ACL Authorization: KafkaAcls визначають що кожен principal може робити: ALLOW user=orders-producer OPERATION=WRITE TOPIC=orders. DENY default. 4) Schema Registry auth: basic auth або OAuth. 5) Network: Kafka у private subnet, тільки через VPC, no public endpoint. Для managed Kafka (Confluent Cloud, MSK): RBAC з roles (Developer, Operator, SystemAdmin).
Що таке data catalog і навіщо він потрібен?
Data catalog = searchable inventory всіх data assets: таблиці, колонки, значення, власники, опис, якість, lineage. Без catalog: нові аналітики не знають що є і де шукати (onboarding 2 тижні замість 2 годин), дублікати — 3 команди роблять одне і те ж, "хто-такий цей user_id?" — ніхто не знає. Основні tools: DataHub (Apache-licenced OSS, LinkedIn, Uber), OpenMetadata (community fork від Amundsen), Amundsen (Lyft, простіший), Alation/Atlan/Collibra (enterprise SaaS). DataHub фічі: column-level lineage з dbt/Spark/Kafka, Slack integration (питаєш "хто власник таблиці X?" → bot відповідає), PII tags, freshness SLAs, assertions. Практика: автоматична індексація через connectors + людська документація для business context.
Encryption at rest vs in transit — що це і як реалізувати?
Encryption at rest: дані зашифровані коли зберігаються. S3: SSE-S3 (AWS managed key) — за замовчуванням з 2023, SSE-KMS (Customer Managed Key через AWS KMS) — ваш ключ, ви контролюєте ротацію і audit, SSE-C (client-side key). Для GDPR/HIPAA — SSE-KMS з CMK. EBS volumes: AES-256 шифрування. RDS/databases: encryption at rest у налаштуваннях. Encryption in transit: TLS 1.2+ для всіх API endpoints, S3 — enforce_tls через bucket policy (deny HTTP), Kafka SSL listeners, PostgreSQL ssl=require. Ключі: ніколи не хардкодити у коді. AWS Secrets Manager або Vault для credentials. KMS для encryption keys. Rotation — автоматична через Secrets Manager (30/90 днів).

Chaos engineering — мінімум

Не повноцінна chaos-practice, але базові експерименти для перевірки resilience:

  • Вбити один Kafka broker pod — замовлення продовжують писатись?
  • Вбити Airflow scheduler — DAGs resume після відновлення?
  • Вбити Spark executor посеред джобу — task re-execute без duplicates?

Спроектуйте real-time аналітичну платформу для e-commerce з 10M events/day
Системний дизайн: Sources (mobile app, web, payment) → Kafka (3 brokers, replication=3, retention 7d) → Flink (real-time aggregations: GMV/5min window by country, fraud detection, session tracking) → Kafka sink topics + Iceberg silver layer. Batch: Airflow DAG щоночі: Iceberg silver → dbt models → gold layer → ClickHouse DWH + BigQuery для ad-hoc. Storage: S3 medallion (bronze/silver/gold), Iceberg для ACID. Metadata: Schema Registry для Avro schemas, DataHub для lineage. Infrastructure: K8s (Spark operator, Airflow на K8sExecutor, Kafka StatefulSet), Terraform, ArgoCD. Моніторинг: Prometheus + Grafana, Kafka consumer lag alerts, dbt tests post-run. 10M events/day = ~115 events/sec → Kafka topic з 6 партиціями, 3 Flink task managers вистачить.
Як би ви оптимізували Spark job що виконується 4 години?
Послідовність: 1) Spark UI → знайти найповільніший stage, 2) Перевірити на skew (один task набагато повільніший), 3) Перевірити shuffle size (якщо >50GB — щось не так з join або groupBy), 4) Перевірити кількість partition (якщо 2 partitions на 200GB → repartition(200)), 5) Перевірити join type — broadcast? sort-merge? cartesian? 6) Column pruning — чи читаємо тільки потрібні колонки? 7) Predicate pushdown — чи фільтри застосовуються до читання? Типові фікси: partition skew → salting або AQE skew join, join без broadcast → перевірити розмір таблиць і підвищити autoBroadcastJoinThreshold, schema inference на CSV → explicit schema definition, collect() on large df → замінити на write + subsequent read, repartition(1) → remove або coalesce(n).
Чим відрізняється Data Engineer від Data Scientist та Analytics Engineer?
Data Engineer: будує та підтримує infrastructure — pipelines, data lakes, streaming, DWH. Мова: Python, SQL, Scala. Знання: distributed systems, Kafka, Spark, K8s. Output: надійні, масштабовані data pipelines. Analytics Engineer: будує data models у DWH (dbt), collaborates with analysts. SQL-heavy. Знання: dimensional modeling, dbt, BI tools. Як боротись з дублікатами у streaming pipeline?
Дублікати у streaming — неминуче: producer retry, consumer at-least-once, network issues. Стратегії: 1) Idempotent Producer (Kafka): enable.idempotence=true — Kafka дедуплікує на broker рівні (в межах сесії). 2) Unique constraint у sink: INSERT INTO ... ON CONFLICT (event_id) DO NOTHING — RDBMS відкине дублікат. 3) Window deduplication у Flink/Spark: DISTINCT by event_id в межах вікна часу (якщо retry приходить протягом 10 хв). 4) Iceberg MERGE: MERGE INTO target USING source ON target.event_id = source.event_id WHEN MATCHED THEN UPDATE SET ... WHEN NOT MATCHED THEN INSERT — upsert семантика. 5) Redis bloom filter: перед INSERT перевірити чи є event_id вже (O(1), ~10MB на 1M IDs). Кращий підхід: event_id генерується на клієнті (UUID v4), є у кожному повідомленні, всі sink-операції ідемпотентні за event_id.
Розкажіть про найскладнішу технічну проблему яку ви вирішили
Це поведінкове питання — але у технічному контексті. Структура відповіді за STAR (Situation, Task, Action, Result): Situation — контекст, що за система, масштаб. Task — яка конкретна проблема виникла (production outage, performance degradation, data quality issue). Action — як ви діагностували: які інструменти використали, як звузили проблему, яке рішення обрали і чому саме воно. Result — результат у цифрах: зменшили latency з 4год до 30хв, зекономили $5k/місяць, знайшли дублікати що спотворювали GMV на 3%. Якщо немає production experience — використовуйте ShopWave: "При розробці capstone project я зіткнувся з..." Важливо: показати систематичний підхід до debugging, а не "пощастило вгадати".
Що таке schema evolution і як її підтримувати у продакшн?
Schema evolution = зміна структури даних не ламаючи існуючих producers/consumers. Типові зміни: add column, rename column, change type, remove column. Safe changes (backward/forward compatible): додати nullable колонку (нові рядки мають значення, старі — null), видалити колонку (старі readers просто ігнорують). Unsafe: rename column (старі readers шукають за старим ім'ям), змінити тип (int → string може зламати aggregations). Schema Registry (Confluent): forward/backward/full compatibility mode — registry відхилить несумісну зміну. Iceberg: column-based ID mapping — перейменування не ломає планів запитів. dbt: on_schema_change='append_new_columns' у incremental моделях. Best practice: завжди backward compatible зміни, версіонування топіків при breaking changes (orders-v2), feature flags для поступового rollout нових полів.
Як виглядає "good data pipeline" з точки зору quality?
Hімецька якість у data pipeline: 1) Idempotency — перезапуск не дублює дані, не пропускає, дає той самий результат. 2) Observability — logging, metrics, alerting. Знаєте про проблему ДО того як бізнес помітив. 3) Data quality tests — dbt tests або Great Expectations: not_null, unique, row count bounds, freshness SLA. 4) Error handling — DLQ для невалідних даних, retry з backoff, circuit breaker для downstream failures. 5) Documentation — source + schema + business meaning. Новий інженер розбирається без вас. 6) Backfill-friendly — pipeline може переобробити будь-який день за --date параметром. 7) Cost-aware — partition pruning, column pruning, incremental processing. 8) Version-controlled — код + config у Git, deployment через CI/CD. Якщо будь-який з пунктів відсутній — pipeline технічно працює, але в enterprise не виживе.

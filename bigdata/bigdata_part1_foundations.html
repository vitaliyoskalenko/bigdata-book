Вступ до Big Data Infrastructure

Big Data — клас інженерних задач, де обсяг, швидкість або різноманітність даних унеможливлюють використання однієї реляційної БД. Розбираємо, де пролягає ця межа, чому вона існує технічно, і як з 2004 року сформувався сучасний Modern Data Stack із десятків інструментів, які навчились між собою ладнати.

Що таке Big Data — визначення через 5V

У 2001 аналітик Дуг Лейні з Gartner запропонував «3V»: Volume, Velocity, Variety. Пізніше додали Veracity та Value, і тепер маємо 5V. Це не магічна межа «з якого розміру дані стають big» — це характеристики задачі, які роблять класичний моноліт (PostgreSQL на одному сервері) недостатнім рішенням.

Аналогія для початківця

Уявіть, що PostgreSQL — це потужний особистий органайзер. Він чудово працює для одного офісу. Big Data — це коли у вас 10 000 офісів по всьому світу, кожен надсилає документи кожну секунду, документи бувають у різних форматах, і деякі з них — відео. Жоден один органайзер не впорається. Треба будувати розподілену бібліотеку з іншими правилами.

V1
Volume

Обсяг, який не поміщається на один сервер. Умовно від сотень GB активних даних або ТБ+ історії. Netflix обробляє ~1.3 ПБ на день.

V2
Velocity

Швидкість надходження. Десятки тисяч подій/сек, real-time обробка. Тригер, що тягне streaming замість batch.

V3
Variety

Структуровані (SQL-таблиці), напівструктуровані (JSON, логи), неструктуровані (відео, аудіо, текст, зображення).

V4
Veracity

Якість та довіра до даних. Дублі, пропуски, неузгодженості, затримки. 80% часу інженера — про це.

V5
Value

Чи приносять дані бізнес-цінність. Найважливіше V — без value всі інші безглузді. Data ROI.

V1 Volume — аналогія

«Бібліотека Вернадської» — 15 млн книг, але весь шлях потрібен лише стелаж «Ки», не вся будівля. Object storage + column pruning — те саме.

V2 Velocity — аналогія

Банківський термінал відображає баланс одразу після оплати — не «за ніч». Velocity — це коли бізнес не може чекати до ранку.

V3 Variety — аналогія

Замість таблиць Excel (SQL) уявіть: PDF-чеки, голосові повідомлення, GPS-трек телефону і логи сервера — все одночасно. Ось Variety.

V4 Veracity — аналогія

80% часу Data Engineer — прибирання «сміття»: дублікати замовлень, пропущені дати, негативні ціни. Veracity — це впевненість, що цифрам можна довіряти.

Як масштабувати — покрокова стратегія

Не перестрибуйте через рівні. Кожен крок — реакція на конкретний bottleneck:

1
PostgreSQL → виносимо аналітику окремо

Prod PostgreSQL починає гальмувати від аналітичних запитів. Додаємо read-replica → перенаправляємо Metabase туди. Або піднімаємо окремий BigQuery/ClickHouse і робимо нічний dump.

Тригер: prod DB > 70% CPU від аналітичних запитів
2
ETL скрипти → Airflow + S3 data lake

Нічні cron-скрипти починають падати і не мають retry. Piднімаємо Airflow (Cloud Composer або self-hosted), всі raw дані пишемо в S3/GCS у Parquet. Spark batch для трансформацій. Перший справжній Data Engineer milestone.

Тригер: > 10 ETL-скриптів, часті fails без алертів
3
Batch → Kafka + streaming (Lambda архітектура)

Бізнес хоче fraud detection або live dashboards — batch раз на добу не підходить. Додаємо Kafka як центральну шину: всі сервіси пишуть events у Kafka. Flink або Spark Structured Streaming рахує метрики в реальному часі. Batch pipeline залишається для historical аналізу.

Тригер: потрібна латентність < хвилини або real-time алерти
4
Монолітний lake → Data Mesh (domain-driven)

Центральна data team не встигає обробляти запити від 5+ product teams. Переходимо на Data Mesh: кожна команда (payments, logistics, marketing) відповідає за свої data products з власними SLA і schema contracts. Federated governance через DataHub з central policies.

Тригер: > 3 команди, data team — bottleneck для бізнесу

Питання на співбесіді — Модуль 1
01Що таке Big Data і коли PostgreSQL перестає вистачати?
+
Немає точного порогу — Big Data це характеристики задачі, а не розмір. PostgreSQL перестає вистачати коли: (1) дані не поміщаються на 1 диск — Volume; (2) потік надходження виміряється в тисячах записів/сек — Velocity; (3) дані різного формату (JSON, відео, логи) — Variety; (4) SLA не можна виконати горизонтально масштабуючи один вузол. Практично: якщо Full-table scan займає >10 сек і оперативна відповідь критична — час дивитись на інші рішення.
02Lambda vs Kappa — поясніть різницю та коли обрати кожну.
+
Lambda: дві паралельні гілки — batch (точна, повільна) + speed (наближена, швидка). Serving layer поєднує. Плюс: точність + low latency. Мінус: дубль логіки в двох стеках, складно тримати синхронно. Kappa: тільки stream, batch = replay стріму. Простіше, але Kafka потребує довгого retention. У 2026 часто обирають Modern Data Stack з lakehouse (Iceberg/Delta) — він закриває більшість кейсів без складності Lambda.
03Чим відрізняється Data Warehouse від Data Lake і Data Lakehouse?
+
DWH (Snowflake, BigQuery): структуровані очищені дані, schema-on-write, ACID, дорогий, ідеальний для BI. Data Lake (S3 + Parquet): все підряд, schema-on-read, дешевий, гнучкий, але без ACID і повільний для SQL. Lakehouse (S3 + Iceberg/Delta): найкраще з обох — open format + ACID + time travel + engine-agnostic. У 2026 — дефолтний вибір для нових платформ.
04Назвіть 5V і поясніть яке з них найважливіше для бізнесу.
+
Volume (обсяг), Velocity (швидкість), Variety (різноманітність), Veracity (якість), Value (цінність). Найважливіше — Value. Решта чотири — це технічні виклики, але якщо дані не створюють бізнес-цінності, вони нічого не варті. Bagато компаній колектують петабайти і не знають навіщо. Починайте з питання «яке рішення ці дані дозволяють приймати краще?»
05Що таке Medallion Architecture і навіщо три шари?
+
Bronze: raw-дані 1:1 з джерелом, append-only — ніколи не видаляємо, завжди можна перерахувати. Silver: очищено, типізовано, дедуплікований — «правда про факти». Gold: агреговані бізнес-метрики, готові для BI. Навіщо три? Щоб можна було перерахувати будь-який шар з попереднього без звернення до джерела. Помилка в Silver → перерахуємо з Bronze, не чіпаючи джерело.
06Чим Data Engineer відрізняється від Analytics Engineer і Data Scientist?
+
Data Engineer: будує інфраструктуру — pipelines, storage, streaming. Spark, Kafka, Kubernetes. Analytics Engineer: dbt, SQL-моделювання, метрики — між DE і аналітиком. Споживає те, що DE побудував, але сам трансформує у бізнес-логіку. Data Scientist: ML-моделі, статистика, A/B — споживає curated-шар. У малих компаніях одна людина робить все, у великих — чітке розмежування.
07Чому MapReduce пішов з ужитку? Що прийшло на зміну?
+
MapReduce скидав проміжні результати на диск після кожного кроку. 10-кроковий pipeline = 10 читань/записів HDFS = повільно. Spark вирішив це in-memory обчисленнями та lazy DAG execution — проміжні дані в RAM executors, на диск тільки при шаффлі або явному persist. Результат: 10-100× швидше на ітеративних workloads (ML). Spark став стандартом ~2015-2020. Зараз ще Flink (streaming), Trino/DuckDB (interactive SQL).

Системи зберігання даних

HDFS (Hadoop Distributed File System) — розподілена файлова система, що ділить файли на блоки по 128 MB і реплікує їх на 3 вузли кластеру. Amazon S3 і сумісні object storage (GCS, Azure Blob) — хмарна альтернатива: необмежений обсяг, плата за GiB, але без POSIX-семантики. Поверх цих сховищ формуються три архітектурних підходи — Data Lake, Data Warehouse, Data Lakehouse — кожен зі своїм компромісом між гнучкістю, швидкістю і надійністю.

HDFS — класика, яку треба знати

Hadoop Distributed File System був стандартом 2008-2018. Зараз його повільно витісняє object storage, але: (а) легасі-системи досі живі, (б) принципи HDFS пояснюють, як взагалі працюють distributed storage, (в) це обов'язкове питання на співбесіді.

HDFS архітектура · NameNode + 3 DataNodes · rack-aware replication · (наведіть на блок для опису)
  • NameNode тримає metadata: які файли, розбиті на які блоки, на яких DataNodes. Critical — якщо падає, весь кластер лягає. Тому є Secondary / Standby NameNode (HA через Quorum Journal Manager).
  • DataNodes тримають сирі блоки (за замовчанням 128 MB). Реплікація — 3 копії на різні вузли, бажано на різних rack-ах.
  • Rack awareness — HDFS знає топологію мережі, щоб одна копія була на іншому fizичному стелажі. Захист від падіння rack-а цілком.
  • YARN — менеджер ресурсів у тій самій екосистемі. Управляє, які джоби на яких DataNodes виконуються (data locality).
Плюси HDFS
  • Data locality — обчислення поруч з даними
  • Дешеве сховище для великих файлів on-prem
  • Зріла екосистема (Spark, Hive, HBase)
  • Контроль — все своє, ніяких хмар
Мінуси HDFS
  • Треба самому тримати залізо, NameNode, backup
  • Погано з «маленькими файлами» (< 128 MB кожен)
  • Нема поділу compute/storage — масштабується разом
  • Складний HA; вразливість на NameNode
  • У 2026 — повільно витісняється object storage

Партиціонування — як не зробити «lake of small files»

Неправильно розкладені дані можуть перетворити S3 на повільне болото. Основні принципи:

  • Партиціонуйте за тим, за чим фільтруєте. Найчастіше це дата: year=2026/month=04/day=21/. Query engine (Spark, Athena, Trino) зможе пропустити непотрібні папки (partition pruning).
  • Не дробіть занадто. Партиція по хвилинах з 10 файлами по 1 KB — катастрофа. Цільовий розмір файла — 128-512 MB.
  • High-cardinality поля — погані партиції. Партиція за user_id створить мільйони папок.
  • Компакція. Регулярно перепаковуйте «дрібноту» у великі файли. Delta/Iceberg уміють це автоматично.
  • Bucketing — додатковий поділ всередині партицій за хешем ключа для ефективних joins.

# Читання Parquet з S3 через pandas
def read_parquet_from_s3(bucket: str, key: str) -> pd.DataFrame:
    obj = s3.get_object(Bucket=bucket, Key=key)
    return pd.read_parquet(BytesIO(obj['Body'].read()))

# Запис DataFrame у Parquet → S3 (ZSTD стиснення)
def write_parquet_to_s3(df: pd.DataFrame, bucket: str, key: str):
    buffer = BytesIO()
    df.to_parquet(buffer, engine='pyarrow', compression='zstd', index=False)
    buffer.seek(0)
    s3.put_object(Bucket=bucket, Key=key, Body=buffer.getvalue())

# Партиціонований запис (year/month/day структура)
def write_partitioned(df: pd.DataFrame, bucket: str, base_path: str):
    for (year, month), group in df.groupby(['year', 'month']):
        key = f"{base_path}/year={year}/month={month:02d}/data.parquet"
        write_parquet_to_s3(group.drop(columns=['year', 'month']), bucket, key)

# Автоматична lifecycle policy — переводить у Glacier через 90 днів

Під капотом: як великі дані зберігаються і доставляються

Знати «що робить HDFS» недостатньо для розуміння системи. Тут пояснимо чому системи влаштовані саме так — через алгоритми зберігання, принципи масштабування та обмеження CAP теореми.

① B-Tree vs LSM Tree — два підходи до зберігання

Усі системи зберігання даних будуються на одній з двох фундаментальних структур. Вибір між ними визначає, для чого система підходить краще.

B-Tree vs LSM Tree · два фундаментальних підходи до зберігання даних · (наведіть для пояснення)
② Consistent Hashing — як дані розподіляються між вузлами

Уявіть ring з числами 0..2³² по колу. Кожен вузол кластера займає позицію на ring (через hash від IP/ID). Кожен ключ даних теж хешується і потрапляє до ближнього за годинниковою стрілкою вузла.

⊕
Проблема звичайного hashing

node = hash(key) % N — при зміні N (додали/видалили вузол) ≈ 100% ключів перерозподіляються. На кластері з ТБ даних це катастрофа.

◯
Consistent hashing — рішення

При додаванні вузла перерозподіляється тільки ≈ 1/N частина ключів (від попереднього вузла до нового). Cassandra, DynamoDB, Redis Cluster — всі використовують цей принцип.

↔
Virtual nodes (vnodes)

Кожен фізичний вузол займає кілька позицій на ring (128+ vnode). Забезпечує рівномірний розподіл навіть коли вузли різної потужності. Cassandra: за замовчуванням 256 vnode/вузол.

🔑
Partitioning key вибір

Від partition key залежить де лежать дані. Погано: user_status (hot partition — 90% users = active). Добре: user_id (рівномірний розподіл через хеш).

③ CAP теорема — фундаментальне обмеження розподілених систем

Теорема Брюера (2000): розподілена система може одночасно гарантувати лише два з трьох властивостей.

ВластивістьЩо означаєСистеми що жертвують нею
C — Consistency Кожен читач бачить останній запис або помилку. Всі вузли відповідають однаковими даними. DynamoDB, Cassandra (eventual consistency за замовчуванням)
A — Availability Кожен запит отримує відповідь (не обов'язково останні дані). Система не повертає timeout/error. HBase, Zookeeper, etcd (жертвують A при network partition)
P — Partition Tolerance Система продовжує роботу при обриві мережі між вузлами (network partition). Ніхто — в реальних розподілених системах P є обов'язковим
Практичне правило

Оскільки P (partition tolerance) — неминучий у будь-якій реальній мережі, вибір насправді між CP vs AP: або відмовляємо в обслуговуванні при збої (CP), або відповідаємо застарілими даними (AP). Для Big Data систем: HDFS/HBase = CP, Cassandra = AP (конфігурується через ConsistencyLevel), Kafka = CP (partition leader — єдине джерело правди).

⑤ Write-Ahead Log (WAL) і Bloom Filter
📋
Write-Ahead Log (WAL)

Перед будь-якою зміною даних — спочатку записуємо намір у лог (sequential write на диск). Якщо система впала — при відновленні читаємо WAL і повторюємо операції. Використовується у PostgreSQL, Kafka (commitlog), HDFS edit log, HBase HLog. Гарантує durability навіть при crash.

🌸
Bloom Filter

Probabilistic structure: швидко відповідає «точно немає» або «можливо є». Розмір: 10 байт на мільйон ключів! Cassandra і RocksDB тримають Bloom Filter для кожного SSTable — перед читанням перевіряємо фільтр: якщо «точно немає» → не відкриваємо файл взагалі. False positive rate < 1%.

🗜️
Compaction стратегії

LSM-системи накопичують SSTable-файли. Compaction об'єднує їх: видаляє tombstones (deleted records), merge sorted runs → менше файлів = швидше читання. Стратегії: SizeTiered (Cassandra default, агресивний), Leveled (LevelDB/RocksDB, менше read amplification), TimeWindow (Cassandra для time-series).

📊
Column Store vs Row Store

Row store (PostgreSQL): один рядок — поряд на диску. Добре для OLTP (читаємо весь рядок). Column store (Parquet, ClickHouse): одна колонка — поряд. Добре для OLAP (SELECT amount, country — читаємо тільки 2 колонки з 50). Compression 5-10× краще в column store (схожі значення поряд).

Обмеження і ризики, які треба знати
  • HDFS NameNode bottleneck: весь namespace у RAM одного процесу. При 100M+ файлів — NameNode стає вузьким місцем по пам'яті (кожен файл ≈ 150 bytes метаданих). Рішення: Federation, namespace quotas.
  • Kafka partition count: більше партицій ≠ завжди краще. 10 000 партицій на брокер → вузьке місце при відновленні після перезапуску (leader election для кожної). Правило: max 4000 партицій / брокер.
  • S3 rate limiting: за замовчуванням 3500 PUT/s і 5500 GET/s на prefix. При Spark з тисячами виконавців — можливий 503 SlowDown. Рішення: розподіляти файли по різних prefix-ах (не всі у year=2026/).
  • Small files problem: мільйон файлів по 1 KB — катастрофа для HDFS (NameNode RAM) і S3 (LIST API calls). Рішення: compaction, Iceberg/Delta auto-compaction, Spark coalesce перед записом.
  • Clock skew у distributed systems: різний час на серверах → неправильний порядок подій. Kafka, Flink використовують event-time + watermarks, не processing time. NTP синхронізація ≤ 10ms — обов'язкова.

Питання на співбесіді — Модуль 2
01Як HDFS зберігає файли? Що таке NameNode і DataNode?
+
HDFS ділить файл на блоки (128 MB за замовчанням) і розкладає їх по DataNodes з replication factor 3 (rack-aware). NameNode тримає metadata: де які блоки, на яких нодах. Один на кластер — single point of failure, тому є Standby NameNode через QJM. DataNode зберігає сирі блоки і регулярно посилає heartbeat до NameNode. NameNode відповідає клієнту «твій файл блоки 1,2,3 на DataNode-7,3,12» — клієнт читає напряму з DataNode.
02Чому object storage (S3) витісняє HDFS у нових системах?
+
Ключова відмінність: розділення compute і storage. У HDFS вони зчеплені — масштабуєш одне, масштабуєш інше. S3 масштабується нескінченно незалежно. Інші причини: не треба підтримувати кластер залізо; 11 дев'яток надійності «з коробки»; дешевший; S3 API став галузевим стандартом — Spark, Trino, Flink, DuckDB підтримують нативно. HDFS залишається у legacy on-premise системах та де критична data locality.
03Як правильно обрати ключ партиціонування? Назвіть антипатерни.
+
Правила: (1) партиціонуйте за тим, за чим фільтруєте найчастіше — зазвичай дата; (2) цільовий розмір партиції — 128 MB–1 GB після стиснення; (3) уникайте high-cardinality ключів (user_id, order_id → мільйони папок). Антипатерни: партиція по хвилинах (забагато маленьких файлів), партиція по user_id (кардинальність мільйони), відсутність компакції (накопичення малих файлів від streaming).
04Що таке «small files problem» і як його вирішити?
+
Проблема: мільйони файлів по 1-10 KB в S3/HDFS. Симптоми: Spark витрачає більше часу на metadata lookup ніж на обчислення, S3 LIST запити коштують грошей і часу, NameNode в HDFS перевантажений. Рішення:
  • Компакція (compaction) — регулярно склеювати маленькі файли у великі (128-512 MB)
  • Збільшити batch interval у streaming
  • Delta/Iceberg auto-compaction OPTIMIZE
  • Spark coalesce() перед записом
05S3 Standard vs S3-IA vs S3 Glacier — коли що обирати?
+
Standard ($0.023/GB): активні дані, запити кожен день — поточний місяць, hot-партиції. Infrequent Access ($0.0125/GB): читають рідко але швидко потрібно — від 30 днів до 1 року. Є retrieval fee $0.01/GB. Glacier Instant ($0.004/GB): доступ потрібен рідко, але швидко — 1-5 ms. Glacier Deep Archive ($0.00099/GB): compliance-архіви, відновлення 12 год. Використовуйте S3 Lifecycle для автоматичного переходу між класами.
06Навіщо 11 дев'яток надійності S3 — що це значить на практиці?
+
99.999999999% durability = втрата 1 об'єкта на 100 млрд на рік. AWS досягає цього через автоматичну реплікацію на мінімум 3 AZ в межах регіону. Важливо: надійність ≠ доступність. S3 Standard SLA — 99.9% availability. Якщо потрібна крос-регіональна DR — використовуйте S3 Cross-Region Replication. Версіонування захищає від випадкового видалення, але не від corruption.

Формати даних та серіалізація

Серіалізація — перетворення структурованих даних у байти для зберігання або передачі. Вибір формату визначає стискання, схему і швидкість читання: Parquet і ORC зберігають дані по стовпцях — Spark читає тільки потрібні колонки, ігноруючи решту; Avro і Protobuf — рядково з вбудованою схемою, ідеально для Kafka; JSON і CSV — текст без схеми, зручний для людини, але дорогий для машини. Той самий датасет у Parquet займає в 5–10 разів менше місця і читається в 10–100 разів швидше, ніж у CSV.

Row vs Columnar — суть відмінності

Два способи фізично розкласти таблицю на диску. Відмінність — фундаментальна для продуктивності аналітичних запитів.

Row vs Columnar · чому аналітичні запити ~10× швидші на columnar

Columnar формат дає три ключові виграші для аналітики:

  • Column pruning — читається тільки те, що потрібно. SELECT 3-х колонок з 50 → читаємо 6% диска.
  • Кращий compression ratio — сусідні значення у колонці схожі (наприклад, колонка country має багато повторів «Ukraine»). Алгоритми типу RLE, dictionary encoding, delta стискають це на порядки.
  • Vectorized execution — CPU обробляє пачки однакових типів у SIMD-інструкціях. Для OLTP (одна транзакція — багато полів з одного рядка) — навпаки, row-based краще.

Apache Parquet — детально

Створений у 2013 спільно Twitter і Cloudera. Галузевий стандарт №1 для analytical workloads. Підтримується всіма query engines (Spark, Trino, Athena, BigQuery external tables, DuckDB, pandas, polars).

Структура Parquet · row groups + column chunks + footer зі статистикою

Ключові особливості Parquet:

  • Row groups (~128 MB) — «горизонтальні зрізи». Паралелізм.
  • Column chunks всередині row group — стиск окремо для кожної колонки.
  • Footer з схемою + min/max statistics по кожному column chunk → query engine фільтрує row groups без читання (predicate pushdown).
  • Кодування: dictionary (коли мало унікальних значень), RLE (run-length), delta, bit-packing, PLAIN.
  • Compression: Snappy (швидко, дефолт), GZIP (сильніше, повільніше), ZSTD (оптимум 2024), LZ4.
  • Immutable — файл не редагується, тільки переписується.

JSON, CSV — коли (не) використовувати

  • CSV — тільки для експорту людям. Немає типів, немає схеми, переноси в комах, дужках, лапках ламають усе. Не використовуйте у продакшні.
  • JSON — ok для інтеграцій, webhook-ів, raw-подій (bronze-шар). Для аналітики переводьте у Parquet. JSONL (один JSON на рядок) краще за «один великий JSON».
  • JSON Lines + gzip — прийнятний компроміс для сирих логів, але у 3-5× повільніший за Parquet у запитах.

Schema Registry — навіщо він

Коли 20 сервісів пишуть події в Kafka, треба якось домовитись про формат. Схема не в коді кожного — це хаос. Schema Registry (Confluent) зберігає схеми централізовано, перевіряє compatibility при спробі опублікувати нову версію (BACKWARD, FORWARD, FULL). Використовується з Avro або Protobuf.

import pyarrow as pa
import pyarrow.parquet as pq
import pandas as pd

# Запис Parquet із ZSTD стисненням (найкращий баланс 2026)
df = pd.read_csv('orders.csv')
pq.write_table(
    pa.Table.from_pandas(df),
    'orders.parquet',
    compression='zstd',
    compression_level=3,      # 1-22, дефолт 3 — добрий баланс
    row_group_size=512 * 1024, # 512 KB рядків на row group
    write_statistics=True,     # min/max stats для predicate pushdown
)

# Читання тільки потрібних колонок — column pruning
table = pq.read_table('orders.parquet', columns=['order_id', 'amount', 'country'])
df = table.to_pandas()

# Predicate pushdown — читає тільки потрібні row groups
filters = [('country', '=', 'Ukraine'), ('year', '=', 2026)]
df_ua = pq.read_table('orders.parquet', filters=filters).to_pandas()

# Читання партиціонованого датасету з S3 (через fsspec)
import s3fs
fs = s3fs.S3FileSystem(endpoint_url='http://localhost:9000')
dataset = pq.ParquetDataset(
    's3://shopwave-silver/orders/',
    filesystem=fs,
    filters=[('year', '=', 2026)],
)
df = dataset.read(columns=['order_id', 'amount']).to_pandas()
Polars — сучасна альтернатива pandas (у 5-50× швидше)
# pip install polars
import polars as pl

# Lazy evaluation — будує план, виконує тільки при collect()
result = (
    pl.scan_parquet("s3://shopwave-silver/orders/**/*.parquet")
    .filter(pl.col("country") == "Ukraine")
    .filter(pl.col("year") == 2026)
    .group_by(["month", "category"])
    .agg([
        pl.sum("amount").alias("total_gmv"),
        pl.n_unique("order_id").alias("order_count"),
        pl.mean("amount").alias("aov"),
    ])
    .sort("total_gmv", descending=True)
    .collect()  # тільки тут виконується
)

# Запис у Parquet
result.write_parquet("gmv_by_month.parquet", compression="zstd")

Schema Evolution — сумісність версій схем

Реальні системи живуть роками: схема таблиці або Kafka-топіку змінюється десятки разів. Schema evolution — здатність читати старі дані новим кодом і навпаки, не ламаючи pipeline.

Backward Compatible

Новий код читає старі дані. Дозволено: додати поле зі значенням за замовчуванням, видалити необов'язкове поле. Заборонено: видалити обов'язкове поле, змінити тип (int → string).

✓ Споживач оновлюється першим
Forward Compatible

Старий код читає нові дані. Дозволено: додати нове поле (старий код ігнорує). Заборонено: видалити поле що старий код очікує, перейменувати поле.

✓ Продюсер оновлюється першим
Full Compatible

Обидва напрямки. Дозволено тільки: додати поле з default. Gold standard для production — не обмежує порядок деплою.

✓ Будь-який порядок деплою

Питання на співбесіді — Модуль 3
01Чому Parquet — стандарт для аналітики, а Avro — для streaming?
+
Parquet — columnar: читає тільки потрібні колонки, відмінне стиснення, predicate pushdown. Ідеально для аналітичних запитів типу SELECT AVG(amount). Але не призначений для streaming — immutable, немає schema evolution. Avro — row-based: дуже швидкий sequential write, схема вбудована в файл або Schema Registry, повна schema evolution (forward/backward). Ідеально для Kafka — кожне повідомлення незалежне. Типовий flow: Avro в Kafka → S3 Bronze → Parquet у Silver.
02Що таке predicate pushdown і як Parquet його реалізує?
+
Predicate pushdown = фільтрація якомога ближче до джерела, без зчитування всіх даних. Parquet зберігає min/max statistics для кожного column chunk у footer файлу. Query engine (Spark, DuckDB, Athena) читає тільки footer → якщо WHERE date = '2026-04-01', а row group має min=2024-01-01 та max=2025-12-31 → пропускаємо увесь row group. Типовий результат: читається 5-10% файлу замість 100%. Партиціонування на рівні файлів + predicate pushdown в Parquet = двошаровий захист від зайвого читання.
03Яке стиснення обрати для Parquet? Snappy vs ZSTD vs GZIP.
+
Snappy: швидкість читання/запису, помірне стиснення (~3-5×). Дефолт Spark. Рекомендується коли compute дорожчий за storage. ZSTD (рівень 3): краще стиснення ніж Snappy (~5-8×) при схожій швидкості. Рекомендований вибір у 2026 для нових проектів — кращий баланс. GZIP: найкраще стиснення (~6-10×), але повільніше у 3-5×. Для архівних даних де швидкість не критична. Порада: використовуйте ZSTD рівень 3 для Silver/Gold, Snappy для тимчасових shuffle-файлів.
04Що таке Schema Registry і навіщо він у Kafka-стеку?
+
Без Schema Registry кожен Kafka-producer робить що хоче зі схемою: сьогодні amount — float, завтра — string. Consumer ламається. Schema Registry (Confluent) — централізоване сховище схем Avro/Protobuf/JSON. Producer реєструє схему та отримує schema_id (4 байти) — вставляє в повідомлення замість повної схеми. Consumer по schema_id отримує схему і десеріалізує. Compatibility check: BACKWARD (новий consumer читає старі дані), FORWARD (старий consumer читає нові), FULL — обидва. Запобігає поламаним production pipelines.
05Чому JSON не рекомендується для production аналітичних pipeline?
+
Декілька причин: (1) Розмір — JSON зберігає імена полів у кожному рядку: {"amount":150,"country":"UA"} vs Parquet, де ці назви є тільки раз у schema. Overhead 3-10×. (2) Типи — немає нативних типів для datetime, int64, decimal — все рядок, prone до мовчазних помилок. (3) Швидкість — parsing JSON у Python повільний, немає vectorized execution. (4) Стиснення — JSON погано стискається. Правило: JSON/JSONL — тільки для bronze (raw landing) або API-інтеграцій. Все аналітичне — Parquet.
06Polars vs pandas — коли що обирати у 2026?
+
pandas: зрілий, величезна екосистема, звична синтаксис, більше туторіалів. Добре для <10 GB в пам'яті. Polars: written in Rust, lazy evaluation, ~5-30× швидше pandas на реальних workloads, нативний Parquet/Arrow, паралелізм з коробки. Добре для >1 GB, де pandas повільний. У 2026 для нових data проектів — Polars або DuckDB для intermediate processing, pandas тільки де потрібна sklearn/matplotlib сумісність. Для великих даних (Spark-scale) — PySpark.

Від MapReduce до Spark — навіщо взагалі

MapReduce (Hadoop) обробляє дані у два кроки: map (трансформація рядків) → reduce (агрегація). Проблема: між кожним етапом проміжний результат скидається на диск. Складний pipeline з 10 кроків = 10 читань/записів HDFS. Повільно.

Spark (2010, UC Berkeley AMPLab) переосмислив підхід: in-memory обчислення і DAG execution. Проміжні дані живуть у RAM executors, на диск скидаються тільки за необхідності. Результат — на порядок швидше типове MR-навантаження.

Архітектура Spark

Spark · Driver + Cluster Manager + Executors на Workers
  • Driver — це ваш процес (python/scala), де створюється SparkSession. Керує всім джобом, будує DAG, роздає таски executors.
  • Cluster Manager — виділяє ресурси. У 2026 найпопулярніше — Kubernetes (про це у Частині 3). YARN — легасі.
  • Executors — JVM-процеси на worker-нодах, які виконують tasks і тримають дані в кеші/памʼяті.
  • Task — один «шматок роботи» над однією партицією даних.

Catalyst Optimizer — чому Spark швидший за «ручний» код

Коли ви пишете DataFrame-код або Spark SQL, Spark не виконує його буквально. Запит проходить через 4 фази оптимізації:

  1. Unresolved Logical Plan — парсить SQL/DataFrame операції.
  2. Resolved Logical Plan — звіряє з каталогом таблиць.
  3. Optimized Logical Plan — застосовує rule-based optimizations: predicate pushdown, column pruning, constant folding, join reordering.
  4. Physical Plan — обирає фізичну стратегію (broadcast vs sort-merge join, repartitioning) через cost-based optimizer.

Висновок: пишіть декларативно. Catalyst знає більше за вас.

Transformations vs Actions — lazy evaluation

Transformations (lazy)
  • select, filter, groupBy, join
  • withColumn, drop, orderBy
  • map, flatMap, union
  • Будують DAG, не виконують
Actions (eager)
  • show, collect, count
  • write, save, toPandas
  • take, first, reduce
  • Тригерять виконання DAG

Shuffling — головний ворог продуктивності

Операції типу groupBy, join, distinct змушують Spark переганяти дані між executors по мережі (shuffle). Це дорого: диск + мережа + серіалізація.

  • Broadcast join — якщо одна сторона маленька (<10 MB), її реплікують на всі executors. Без шафлу. Spark часто робить це автоматично.
  • Sort-merge join — дефолт для великих. Обидві сторони сортуються за ключем, потім merge. Це shuffle.
  • Partition skew — коли одна партиція у 100× більша за інші (наприклад, user_id=null). Джоба стоїть на одному executor. Лікується salting або AQE (Adaptive Query Execution у Spark 3+).
  • Bucketing — заздалегідь розкладаєте за хешем ключа → наступні joins уникнуть shuffle.

import argparse

spark = (SparkSession.builder
    .appName("shopwave-bronze-to-silver")
    .config("spark.sql.adaptive.enabled", "true")          # AQE — обов'язково
    .config("spark.sql.adaptive.skewJoin.enabled", "true") # auto skew fix
    .config("spark.hadoop.fs.s3a.endpoint", "http://minio:9000")
    .config("spark.hadoop.fs.s3a.path.style.access", "true")
    .getOrCreate())

spark.sparkContext.setLogLevel("WARN")

# --- 2. Явна схема = швидше читання, раніше помилки ---
order_schema = StructType([
    StructField("order_id",    StringType(),    nullable=False),
    StructField("user_id",     StringType(),    nullable=False),
    StructField("amount",      DoubleType(),    nullable=True),
    StructField("country",     StringType(),    nullable=True),
    StructField("created_at",  TimestampType(), nullable=False),
    StructField("status",      StringType(),    nullable=True),
])

# --- 3. Читання з Bronze (JSON) ---
parser = argparse.ArgumentParser()
parser.add_argument("--date", required=True)  # напр. 2026-04-21
args = parser.parse_args()

bronze_path = f"s3a://shopwave-bronze/orders/event_date={args.date}/"
df_raw = spark.read.schema(order_schema).json(bronze_path)

# --- 4. Трансформації Silver ---
df_clean = (df_raw
    # Фільтр поганих рядків
    .filter(F.col("order_id").isNotNull())
    .filter(F.col("amount") > 0)
    .filter(F.col("created_at").isNotNull())
    # Дедуплікація за order_id — беремо найновіший запис
    .withColumn("_rank", F.row_number().over(
        Window.partitionBy("order_id").orderBy(F.col("created_at").desc())
    ))
    .filter(F.col("_rank") == 1)
    .drop("_rank")
    # Нормалізація country — uppercase, trim
    .withColumn("country", F.upper(F.trim(F.col("country"))))
    # Додаємо колонки для партиціонування
    .withColumn("event_date", F.to_date(F.col("created_at")))
    .withColumn("year",       F.year(F.col("event_date")))
    .withColumn("month",      F.month(F.col("event_date")))
)

# --- 5. Metrics для alerting (ловити аномалії до запису) ---
total_in  = df_raw.count()
total_out = df_clean.count()
dup_rate  = (total_in - total_out) / total_in * 100
print(f"Input: {total_in:,} | Output: {total_out:,} | Dup rate: {dup_rate:.2f}%")

if dup_rate > 20:  # якщо >20% дублів — щось не так
    raise ValueError(f"Too many duplicates: {dup_rate:.1f}%")

# --- 6. Запис у Silver (Parquet + ZSTD) ---
(df_clean
    .repartition("country")  # оптимальний розмір файлів по країні
    .write
    .mode("overwrite")
    .partitionBy("year", "month", "country")
    .option("compression", "zstd")
    .parquet("s3a://shopwave-silver/orders/")

# Silver → Gold: GMV, AOV, конверсія воронки
df_silver = spark.read.parquet("s3a://shopwave-silver/orders/")
df_events = spark.read.parquet("s3a://shopwave-silver/events/")

# GMV і AOV по країні/дню
df_gmv = (df_silver
    .filter(F.col("status") == "COMPLETED")
    .groupBy("event_date", "country")
    .agg(
        F.sum("amount").alias("gmv"),
        F.count("order_id").alias("orders"),
        F.avg("amount").alias("aov"),
        F.countDistinct("user_id").alias("buyers"),
    )
)

# Воронка конверсії: view → cart → purchase
df_funnel = (df_events
    .groupBy("event_date", "country")
    .agg(
        F.sum(F.when(F.col("event_type") == "product_view", 1).otherwise(0)).alias("views"),
        F.sum(F.when(F.col("event_type") == "add_to_cart", 1).otherwise(0)).alias("carts"),
        F.sum(F.when(F.col("event_type") == "purchase", 1).otherwise(0)).alias("purchases"),
    )
    # Broadcast join для невеликого df_gmv (country lookup)
    .join(F.broadcast(df_gmv), ["event_date", "country"], "left")
    .withColumn("conversion_rate", F.col("purchases") / F.col("views") * 100)
)

df_funnel.write.mode("overwrite").parquet("s3a://shopwave-gold/funnel_daily/")

Оптимізація Spark: практичні техніки

BJ
Broadcast Join

Для таблиць <10 MB використовуйте F.broadcast(small_df). Spark реплікує на всі executors — без shuffle. Автоматично при spark.sql.autoBroadcastJoinThreshold=10m.

Ca
Cache / Persist

df.cache() (= MEMORY_ONLY) або df.persist(StorageLevel.MEMORY_AND_DISK). Кешуйте тільки DF, які читаються 2+ разів. Після використання — df.unpersist().

Rp
Repartition vs Coalesce

repartition(N) — повний shuffle, рівномірний розподіл. coalesce(N) — без shuffle, зменшує кількість партицій (тільки вниз). Перед записом — coalesce для меншої кількості файлів.

AQE
Adaptive Query Execution

Spark 3+ автоматично: рекомбінує партиції після shuffle, обирає broadcast/SMJ, виправляє skew. Вмикайте завжди: spark.sql.adaptive.enabled=true.

PP
Partition Pruning

Фільтруйте по партиційних колонках до join/aggregation. Spark не читає непотрібні папки. Перевірте у explain(True) наявність PartitionFilters.

Sa
Salting для Skew

Якщо один ключ займає 80% даних — додайте salt_key = concat(key, rand(10)). Розбиває "жирну" партицію на 10 менших. Потім усуньте salt при агрегації.

Топ-10 помилок, що ламають Spark у prod

Типові помилки початківців
  • collect() на великих даних — df.collect() тягне всі дані в Driver. На 100 GB — OOM. Завжди take(100), show() або агрегати.
  • UDF замість вбудованих функцій — Python UDF виходить з JVM, серіалізується через pickle — у 10-100× повільніше. Шукайте еквівалент у pyspark.sql.functions.
  • schema inference замість явної схеми — Spark читає весь файл для визначення схеми. На 100 GB = зайве повне сканування. Завжди передавайте schema=.
  • persist() без unpersist() — закешовані DF займають пам'ять executors. При багатьох джобах — OOM або евікція кешу в найгірший момент.
  • cartesian product (join без ключа) — df1.crossJoin(df2) або join без умови. N × M рядків. Легко вбиває кластер.
  • count() для перевірки завершення — if df.count() > 0 тригерить повне сканування. Використовуйте df.isEmpty або df.limit(1).count().
  • repartition(1) для одного файлу — спочатку весь shuffle, потім один executor пише. Повільно. Краще: coalesce(1) або просто дозвольте кілька файлів.
  • Ігнорування skew — одна партиція у 100 GB поки решта по 100 MB. Stage "висить" на 1 executor. Симптом: Spark UI показує 99% tasks done, 1 task running.
  • Читання CSV без схеми на prod — CSV не має типів, schema inference повільна і ненадійна. Один файл з "1,234" замість "1234" — вся колонка стає string.
  • Driver memory занадто мала — збираєте результат у Driver (show, collect, toPandas). Для join explain-ів і broadcast це нормально, але для даних — ні. Налаштуйте spark.driver.memory=4g.
🎯 Interview Q&A — Модуль 4: Apache Spark
Що таке lazy evaluation у Spark і чому це важливо?
Spark не виконує трансформації (filter, map, groupBy) одразу — він будує граф операцій (DAG). Виконання починається тільки при action (collect, write, show). Це дозволяє Catalyst Optimizer оптимізувати весь план: переставити фільтри ближче до джерела (predicate pushdown), об'єднати кілька операцій в одну стадію (pipelining), вибрати broadcast join замість sort-merge якщо одна таблиця мала. Без lazy evaluation кожна трансформація виконувалась би окремо — дорого і неефективно.
Чому shuffle — найдорожча операція? Як її мінімізувати?
Shuffle = переміщення даних між executor-ами через мережу. Він потрібен для wide transformations (groupBy, join, distinct, repartition). Дорого через: 1) серіалізація/десеріалізація даних, 2) запис на диск (spill), 3) мережевий трафік. Мінімізація: a) broadcast join замість sort-merge для малих таблиць (<10MB за замовчуванням, можна збільшити spark.sql.autoBroadcastJoinThreshold), b) фільтруй якомога раніше до join/groupBy, c) використовуй AQE — він автоматично перемикає на broadcast якщо build-side виявилась маленькою під час виконання, d) уникай unnecessary repartition.
Broadcast join vs Sort-Merge Join — коли що?
Broadcast join: одна таблиця цілком розсилається на всі executor-и, join виконується локально без shuffle. Ідеально: dimension-таблиці (<200-500MB). Обмеження: велика таблиця не влізе в пам'ять driver + executor. Sort-Merge Join: обидві таблиці сортуються і партиціонуються за join-ключем (shuffle), потім merge локально. Підходить для великих таблиць будь-якого розміру. Дорожче через shuffle. AQE може динамічно переключити SMJ → Broadcast під час виконання якщо реальний розмір build-side виявився < порогу.
Що таке partition skew і як з ним боротись (salting)?
Skew = нерівномірний розподіл даних між партиціями. Одна партиція може мати 90% даних — її executor виконує роботу в 10x довше, решта чекають. Симптоми в Spark UI: один task набагато повільніший за інші у stage. Salting-техніка: до skewed join-ключа додається випадковий суфікс (salt 0-N), таблиця-large партиціонується by (key, salt), таблиця-small репліцюється N разів з різними salt-ами. Тепер дані рівномірно розподілені. AQE 3.0+ підтримує Skew Join Optimization автоматично через spark.sql.adaptive.skewJoin.enabled=true.
DataFrame vs RDD — коли ще потрібні RDD?
DataFrame — основний API. Він має схему, проходить через Catalyst Optimizer і Tungsten engine (ефективна серіалізація). RDD — низькорівневий API без схеми і без оптимізацій. DataFrame у 5-10x швидший ніж RDD для типових операцій. RDD потрібен тільки коли: 1) потрібна точна гарантія порядку елементів, 2) складна кастомна логіка яку не можна виразити через DataFrame API, 3) не-структуровані дані без схеми. У новому коді RDD практично не використовується.
cache() vs persist() — різниця і коли використовувати?
cache() = persist(StorageLevel.MEMORY_AND_DISK). persist() дозволяє вибрати storage level: MEMORY_ONLY (найшвидший, дані губляться при spill), MEMORY_AND_DISK (за замовчуванням для cache), DISK_ONLY, OFF_HEAP (Tungsten). Кешувати варто коли DataFrame буде використовуватись >1 разу в різних операціях. Обов'язково unpersist() після використання — інакше займає пам'ять executor-ів. Кешування не допомагає якщо DataFrame великий і весь spill-ується на диск — тоді краще просто записати в Parquet і перечитати.
Що таке AQE (Adaptive Query Execution) і які оптимізації він робить?
AQE (Spark 3.0+) переоптимізовує план під час виконання на основі runtime-статистики. Три ключові оптимізації: 1) Dynamically coalescing shuffle partitions — після shuffle об'єднує маленькі партиції щоб зменшити overhead (spark.sql.adaptive.coalescePartitions.enabled), 2) Dynamically switching join strategies — якщо build-side виявилась малою, перемикається з SMJ на Broadcast join, 3) Dynamically optimizing skew joins — розбиває великі skewed партиції і реплікує відповідні партиції іншої сторони. AQE вимкнений за замовчуванням до Spark 3.2, з 3.2 — увімкнений. Завжди вмикайте: spark.sql.adaptive.enabled=true.
Як підходити до debugging повільного Spark-джобу?
Послідовність: 1) Spark UI → вкладка Jobs → знайти найдовший job, 2) Jobs → Stages — знайти stage з найбільшою Duration, 3) Stage detail → Tasks — шукати outlier task (на порядок повільніший за медіану = skew), 4) Stage detail → перевірити Shuffle Read/Write Size — якщо shuffle >10GB = неефективний join або відсутній фільтр, 5) Environment → перевірити конфігурацію, 6) SQL → Query Plan tab — шукати BroadcastNestedLoopJoin замість BroadcastHashJoin або CartesianProduct. Найчастіші причини: skew, cartesian product, collect на великих даних, schema inference на CSV, занадто мало партицій після coalesce(1).

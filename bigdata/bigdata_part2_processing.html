Stream Processing — Kafka + Flink

Apache Kafka — розподілений журнал подій (distributed log): producer пише повідомлення, вони зберігаються в незмінному впорядкованому лозі, consumer читає у своєму темпі — хоч через секунду, хоч через тиждень. Apache Flink — stateful stream processor: читає з Kafka і обчислює результати (агрегати, anomaly detection, join-и) з latency від мілісекунд. Разом вони дають відповідь «що відбувається прямо зараз» — антифрод, surge pricing, live-дашборди — те, що batch ETL фізично не може.

Batch vs Streaming — ментальна модель

Batch — це «фотографія»: дані фіксовані, ми їх цілком обробляємо. Streaming — це «відео»: потік нескінченний, треба обробляти по мірі надходження, з обмеженими ресурсами пам'яті.

Apache Kafka — центральна нервова система даних

Kafka з 2011 — де-факто стандарт для event streaming. Це не тільки message queue: це розподілений append-only журнал, з якого багато споживачів можуть читати події незалежно одне від одного. У великих компаніях Kafka — «центр усесвіту даних»: до нього підключені 100+ сервісів.

Kafka · брокери, топіки, партиції з реплікацією, consumer groups

Ключові концепції Kafka

Topic

Логічне ім'я потоку подій. Наприклад, orders, user_clicks, payment_events. За аналогією — «таблиця».

Partition

Фізичний поділ топіка для масштабування. Кожна партиція — окремий append-only лог на одному брокері. Порядок гарантується в межах партиції.

R
Replication

Кожна партиція має replication.factor (типово 3) копій на різних брокерах. Одна — leader, решта — replicas.

Offset

Номер повідомлення у партиції. Монотонно зростає. Споживач сам трекає, де він зупинився.

Consumer Group

Група споживачів-«клонів» ділить партиції між собою. Одна партиція — один активний consumer у групі (для порядку).

K
Message Key

Визначає, в яку партицію потрапить подія (hash(key) % partitions). Повідомлення з однаковим ключем завжди ідуть у ту саму партицію = в тому ж порядку.

Kafka Connect і Schema Registry

Сирий Kafka — це тільки брокер. Для production-стеку додають:

  • Kafka Connect — декларативний фреймворк для source/sink-конекторів. CDC з Postgres через Debezium, sink у S3 через S3Sink, в Elasticsearch, Snowflake — все через JSON-конфіг, без коду.
  • Schema Registry (Confluent) — централізоване сховище Avro/Protobuf-схем. Producer при надсиланні додає в повідомлення лише schema_id (4 байти) замість усієї схеми. Перевіряє compatibility: чи не зламаєте ви споживачів зміною схеми.
  • ksqlDB — SQL поверх Kafka, для легких streaming-агрегацій без Flink.

Stateful processing, windows, watermarks

Це три концепції, без яких streaming перетворюється на набір неправильних цифр. Вивчіть їх ґрунтовно.

Stateful processing

Чистий stateless-обробник: «отримав подію → трансформував → відправив далі». Stateful: «отримав подію → оновив якийсь накопичений стан (counter, агрегат, профіль користувача) → відправив результат». Kafka Streams і Flink зберігають state локально (RocksDB), роблять snapshot-и у S3 для відновлення після падіння.

Event time vs processing time
  • Event time — коли подія реально сталась (timestamp з самої події, який згенерував клієнт).
  • Processing time — коли наш джоб побачив подію.
  • Ці два часи розходяться: клієнт був офлайн 2 години, батчем відправив 100 подій. Event time — давній, processing time — зараз.
  • Для бізнес-правильних агрегацій треба event time.
Windowing
T
Tumbling

Фіксовані непересічні вікна: [12:00-12:05), [12:05-12:10). Для «виторг за кожні 5 хв».

S
Sliding (Hopping)

Розмір 10 хв, крок 1 хв — вікна перекриваються. Для «ковзного середнього останні 10 хв».

Se
Session

Гнучкі вікна з gap-ом. Коли gap між подіями > 30 хв — нова сесія. Класично для user activity.

G
Global

Одне нескінченне вікно. Закривається тригером (custom logic).

Watermarks — коли закривати вікно

Якщо ми чекаємо «усі події за 12:00-12:05», то коли зупиняємо чекати? Подія з event-time 12:03 може прийти о 12:08 (запізнилась). Watermark — це «ми вважаємо, що подій з event-time раніше X більше не буде». Типово — max_event_time_seen − 5 хвилин. Після того, як watermark перетинає кінець вікна, воно закривається і тригерить обчислення. Пізніші події або відкидаються, або йдуть у side output (late data).

Apache Flink — практичний код

Аналогія: Якщо Kafka — це конвеєр з коробками, то Flink — бригада, яка обробляє кожну коробку в реальному часі, пам'ятає скільки коробок пройшло (state), і може відновитись з будь-якої точки після аварії (checkpoint).

result = (
    stream
    .map(lambda s: json.loads(s), output_type=Types.PICKLED_BYTE_ARRAY())
    .map(lambda o: (o["country"], o["total_amount"]),

)

    .set_bootstrap_servers("kafka:9092")
    .set_record_serializer(
        KafkaRecordSerializationSchema.builder()
        .set_topic("shopwave.gmv_5min")
        .set_value_serialization_schema(SimpleStringSchema())
        .build()
    )
    .build()
)

Stateful processing — fraud detection з keyed state
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream.state import ValueStateDescriptor
from pyflink.datastream.functions import KeyedProcessFunction
from pyflink.common import Types

class FraudDetector(KeyedProcessFunction):
    """Детектор: >5 замовлень від одного user_id за 2 хвилини → alert."""

    def open(self, runtime_context):
        # State зберігається локально (RocksDB) + у checkpoint (S3)
        self.count_state = runtime_context.get_state(
            ValueStateDescriptor("order_count", Types.INT())
        )
        self.window_start = runtime_context.get_state(
            ValueStateDescriptor("window_start", Types.LONG())
        )

    def process_element(self, order, ctx):
        now = order["event_time"]
        count = self.count_state.value() or 0
        win_start = self.window_start.value() or now

        # Якщо вікно 2 хв вийшло → скинути лічильник
        if now - win_start > 2 * 60 * 1000:
            count = 0
            self.window_start.update(now)

        count += 1
        self.count_state.update(count)

        if count > 5:
            # Emit fraud alert
            yield {
                "user_id": order["user_id"],
                "count": count,
                "alert_time": now,
                "type": "RAPID_ORDERS"
            }
            # Зареєструвати cleanup timer через 10 хв
            ctx.timer_service().register_event_time_timer(now + 600_000)

    def on_timer(self, timestamp, ctx):
        # Очистити state після таймера
        self.count_state.clear()
        self.window_start.clear()

# Використання:
fraud_stream = (
    parsed_orders
    .key_by(lambda o: o["user_id"])
    .process(FraudDetector(), output_type=Types.PICKLED_BYTE_ARRAY())

Exactly-once semantics

Три рівні гарантій при обробці потоку:

  • At-most-once — подія може загубитись, але не подвоїтись. Дешево, рідко потрібно.
  • At-least-once — подія не загубиться, але може прийти двічі. Дешево, але вимагає idempotent-обробку.
  • Exactly-once — рівно один раз від початку до кінця (end-to-end). Найдорожче. Досягається через транзакційний commit у Kafka + checkpointing у Flink/Spark + idempotent sink.

COUNTRIES = ["UA", "PL", "DE", "FR", "US"]

def delivery_report(err, msg):
    if err is not None:
        print(f"[ERROR] delivery failed: {err}")

def produce_orders(topic: str, n: int = 10_000):
    for i in range(n):
        order = {
            "order_id":     f"ord-{i:08d}",
            "user_id":      f"usr-{random.randint(1, 10_000):06d}",
            "total_amount": round(random.uniform(5.0, 500.0), 2),
            "country":      random.choice(COUNTRIES),
            "event_time":   int(time.time() * 1000),
        }
        producer.produce(
            topic=topic,
            key=string_serializer(order["user_id"]),   # ключ → детермінована партиція
            value=avro_serializer(

                SerializationContext(topic, MessageField.VALUE)
            ),
            on_delivery=delivery_report,
        )
        if i % 1000 == 0:
            producer.poll(0)          # обробити callbacks

    producer.flush()                  # дочекатись доставки всіх повідомлень
    print(f"Produced {n} events to '{topic}'")

    consumer.close()

Change Data Capture (CDC) — інкрементальна синхронізація з реляційних БД

CDC — патерн захоплення кожної зміни (INSERT / UPDATE / DELETE) з реляційної БД і публікації її у Kafka як події. Замість того щоб щоночі робити full dump таблиці, CDC передає тільки дельту в реальному часі.

Аналогія: CDC — як банківська виписка в реальному часі замість щомісячного звіту. Кожна транзакція одразу відома downstream-системам, а не через добу після пакетної вивантаження.
Log-based CDC

Читає WAL (Write-Ahead Log) PostgreSQL або binlog MySQL — не чіпає продакшн-таблиці. Нульове навантаження на БД, latency <1 сек. Це підхід Debezium.

✓ Best practice
Query-based CDC

Виконує SELECT * WHERE updated_at > last_run за розкладом. Простіше, але не захоплює DELETE, потребує поля updated_at, навантажує БД. Підхід Airbyte, Singer.

⚠ Missed deletes
Trigger-based CDC

DB-тригери пишуть у changelog-таблицю при кожній зміні. Ловить всі операції, але подвоює writes у БД. Застаріло — майже не використовується у сучасних системах.

🎯 Interview Q&A — Модуль 5: Kafka + Flink
Як Kafka досягає high throughput і low latency одночасно?
Кілька механізмів: 1) Sequential I/O — запис append-only, OS cache (page cache) робить sequential reads дуже швидкими, дискова латентність близька до мережевої. 2) Zero-copy — sendfile() syscall передає дані з page cache прямо в мережевий буфер без копіювання в user space. 3) Batching — producer збирає повідомлення в batch (linger.ms + batch.size) і відправляє разом. 4) Compression — ZSTD/LZ4/Snappy компресують batch, зменшуючи мережевий трафік. 5) Consumer parallelism — кожна партиція паралельно читається окремим consumer-ом у групі. Разом це дає 1M+ msg/sec на одному брокері.
Що таке consumer lag і як з ним боротись?
Consumer lag = різниця між останнім offset-ом у партиції (high watermark) та committed offset consumer-а. Показує скільки повідомлень «в черзі» не оброблено. Причини: 1) Consumer повільний (важка обробка), 2) Retention — нові повідомлення надходять швидше, ніж consumer справляється, 3) Consumer завис (exception, OOM). Рішення: a) Збільшити parallelism — більше партицій + більше consumer-ів у групі, b) Оптимізувати processing logic (batch inserts замість по одному), c) Scale out consumer group, d) Перевірити DLQ — чи не застрягли повідомлення через recurring errors. Моніторинг: Kafka consumer lag alert у Grafana через JMX або Kafka Exporter для Prometheus.
Exactly-once в Kafka — як це реалізується?
Три компоненти: 1) Idempotent producer (enable.idempotence=true) — producer отримує PID і sequence number; брокер дедуплікує повторні відправки. 2) Transactions — producer.begin_transaction() / commit_transaction() — атомарний запис у кілька партицій/топіків. 3) Isolated consumers (isolation.level=read_committed) — consumer читає тільки committed транзакції, не uncommitted або aborted. Для end-to-end exactly-once (Kafka → обробка → Kafka) потрібні всі три. З Flink: Flink CheckpointedFunction + KafkaTransactionProducer забезпечують exactly-once через 2-phase commit між Flink checkpoint і Kafka transaction.
Event time vs processing time — чому це важливо?
Processing time — час, коли система обробила подію. Event time — час, коли подія реально сталась (мітка часу в самій події). Розходяться через: мережева затримка, мобільні пристрої офлайн, retry-механізми. Для бізнес-аналітики (GMV за вчора, conversion funnel) треба event time — інакше отримаємо "вчорашні" дані у сьогоднішніх звітах. Watermark вирішує проблему "коли закрити вікно": max(event_time) − allowed_lateness. Наприклад, watermark = max_event_time − 5хв означає "ми вважаємо, що всі події старші за watermark вже надійшли".
Як обрати кількість партицій у топіку?
Немає єдиної формули, але правила: 1) Кількість партицій = максимальний паралелізм consumer group. Якщо 6 партицій — максимум 6 активних consumer-ів. 2) Throughput: якщо потрібно 100 MB/s, а один broker може дати 50 MB/s на партицію — мінімум 2 партиції. 3) Не переборщувати: кожна партиція — файл на диску, file handle в OS. 10k+ партицій на брокері = проблеми. Хороший старт: 3-6 партицій для невеликого топіку, 12-24 для великого. Збільшити пізніше можна, але зміна кількості партицій змінює routing key → hash() mapping, що ламає порядок для ключів.
Flink vs Spark Structured Streaming — коли що обирати?
Flink: true streaming (подія за подією), latency ~мс, stateful processing з RocksDB, SQL API, зрілий event time / watermarks. Обирайте для: trading, fraud detection, IoT реального часу, будь-де де latency < секунди критична. Spark Structured Streaming: micro-batch (пачки 100мс-секунди), простіший якщо вже є Spark-стек, єдиний API для batch+stream. Обирайте для: latency ≥ секунди ОК, команда знає Spark, треба єдиний рантайм. У 2026 Flink виграє у latency і stateful, Spark — у simple ETL stream з Databricks.
Що таке Dead Letter Queue і навіщо він потрібен?
DLQ — окремий топік, куди відправляються повідомлення, які consumer не зміг обробити (parse error, validation failure, downstream timeout). Без DLQ: consumer застряє на "отруєному" повідомленні і lag зростає → весь pipeline стоїть. З DLQ: проблемне повідомлення відкладається, обробка продовжується. Потім DLQ аналізується: a) виправляємо баг і replay-имо повідомлення, b) відкидаємо як невалідні дані. Паттерн: зберігати в DLQ оригінальне повідомлення + headers з помилкою + метаданими (original_topic, original_partition, original_offset, error_message, timestamp). Моніторити consumer lag DLQ-топіка теж важливо — якщо він росте, щось системно зламалось.

Data Lakehouse — Delta · Iceberg · Hudi

Data Lakehouse — підхід, що додає транзакційний metadata-шар поверх звичайних Parquet-файлів у S3. Delta Lake (Databricks), Apache Iceberg (Netflix/Apple) і Apache Hudi (Uber) — три реалізації цього шару. Кожна дає те, чого не має «голий» lake: ACID-транзакції, рядковий UPDATE/DELETE, time travel і schema evolution — без переїзду в окрему базу даних.

Чому «просто Parquet у S3» — недостатньо

У Частині 1 ми розкладали події по папках year=2026/month=04/. Це працює для грубого ETL, але на production ламається в кількох місцях:

Проблеми «голого» lake
  • Немає ACID. Два Spark-джоби одночасно пишуть у ту саму партицію — дані можуть бути частково побачені читачем
  • Немає UPDATE/DELETE. Треба переписати всю партицію, навіть якщо змінився 1 рядок. GDPR «видалити юзера 42» стає катастрофою
  • Schema drift. Хтось додав колонку або змінив тип — читачі ламаються
  • Малі файли. Стрім додає по 1 MB щохвилини — через тиждень у S3 мільйони файлів, запити ледь повзуть
  • Немає time travel. «Покажи дані, як було 2 дні тому до багу» — ніяк
  • Listing expensive. S3 LIST через тисячі партицій — повільно і дорого
Що вирішує lakehouse-формат
  • Транзакційний лог (metadata layer) поверх Parquet
  • ACID на рівні таблиці — у тому числі concurrent writes
  • Upsert (MERGE), DELETE, UPDATE — рядок рівня
  • Schema evolution — add/drop/rename колонок без міграцій
  • Time travel — запит на стан таблиці на дату/версію
  • Automatic compaction — склеювання маленьких файлів
  • Partition evolution — змінити схему партицій без переливки

Як це технічно працює — на прикладі Iceberg

Iceberg тримає поверх Parquet-файлів метаслой з трьох рівнів:

Iceberg · catalog → metadata.json → snapshots → manifest lists → data files
  • Catalog тримає вказівник на актуальний metadata.json (Hive Metastore, AWS Glue, REST Catalog, nessie).
  • metadata.json — схема, partition spec, історія всіх snapshots.
  • Snapshot — один «стан таблиці». Кожна операція (INSERT/UPDATE/MERGE) створює новий snapshot. Старі лишаються на диску ⇒ time travel.
  • Manifest list вказує на manifest-файли, manifest містить список data-файлів з min/max статистикою колонок.
  • Data files — звичайні Parquet на S3. Жодних спеціальних форматів — тому Iceberg-таблицю може читати будь-який engine, який розуміє спеку.

Компакція та maintenance

Лог транзакцій і маленькі файли накопичуються. Без періодичного cleanup-у з часом таблиця деградує. Операції, які треба робити регулярно (часто — автоматично або через Airflow DAG):

  • Compaction / Optimize — склеювати дрібні файли у великі (~256 MB). В Iceberg — rewrite_data_files, в Delta — OPTIMIZE.
  • Expire snapshots — видаляти старі snapshot-и з метаданих (вони все одно на диску до vacuum). В Iceberg — expire_snapshots.
  • Remove orphan files — файли, на які більше нема посилань у metadata (з невдалих commit-ів). remove_orphan_files.
  • Vacuum / Physical delete — фізично видалити файли, які більше не потрібні навіть для time travel (Delta VACUUM, Iceberg — через expire).
  • Z-ordering / Sort — перевпорядкувати рядки за hot-колонками для кращого pruning на читанні.

🎯 Interview Q&A — Модуль 6: Data Lakehouse
Чому Delta Lake, Apache Iceberg і Hudi — це не просто Parquet з extras?
Вони вирішують фундаментальну проблему: S3/HDFS — файлові системи без транзакцій. Parquet — формат файлу, а не таблиці. Lakehouse-формати додають metadata layer: Iceberg — metadata.json + manifest lists + manifests; Delta — _delta_log; Hudi — .hoodie. Цей шар забезпечує: 1) Atomicity — операція або повністю виконана або ні (ніяких «напів-записаних» партицій), 2) Isolation — читачі не бачать незавершені записи, 3) Time travel через незмінність snapshot-ів, 4) Concurrent writes з optimistic concurrency control. Iceberg і Delta — найпопулярніші у 2026; Hudi — ніша з сильним upsert CDC.
Iceberg vs Delta Lake — ключові відмінності?
Delta Lake: створений Databricks, OSS з 2019. Тісна інтеграція зі Spark та Databricks. Delta Sharing для cross-org обміну. Простіший у налаштуванні в Databricks-екосистемі. Apache Iceberg: Open standard, підтримується Apache Software Foundation. Незалежний від compute-движка: Spark, Trino, Flink, DuckDB, StarRocks всі читають нативно. Partition evolution — можна змінити схему партицій без переливки даних. Hidden partitioning — дані партиціонуються автоматично, запити не потребують знання схеми партицій. REST Catalog — стандартний API для будь-якого catalog (Polaris, Unity). Iceberg краще для multi-engine, Delta — для Databricks. На 2026 Iceberg виграє у нейтральності.
Що таке compaction і чому він критичний для streaming workloads?
Streaming writes створюють малі файли: Flink пише в Iceberg кожні 60 секунд = 1 MB файл/хвилину = 1440 файлів/добу. Читання 1000 малих файлів набагато повільніше ніж читання 10 великих (overhead на S3 LIST, metadata, відкриття файлу). Compaction = перезапис багатьох маленьких файлів в кілька оптимальних (128-256 MB). Iceberg: CALL system.rewrite_data_files() з target_file_size_bytes. Delta: OPTIMIZE table. Hudi: clustering. Compaction потрібно запускати регулярно (Airflow щодня вночі) або тригерно (після N нових файлів). Без compaction запити деградують з часом: 7 хв → 45 хв за 3 місяці.
Як реалізувати time travel запит в Iceberg?
Два способи: 1) За snapshot ID: SELECT * FROM table VERSION AS OF 12345678 або spark.read.option("snapshot-id", "12345678").table("table"). 2) За timestamp: SELECT * FROM table TIMESTAMP AS OF '2026-03-15 00:00:00' або spark.read.option("as-of-timestamp", timestamp_ms).table("table"). Use cases: аудит — "покажи що змінилось після batch-джобу", дебаг — "що було до помилкового UPDATE", regulatory — "стан бази на кінець місяця для звіту". Важливо: snapshot-и зберігаються тільки в межах retention. Expire snapshots видаляє їх — тому time travel обмежений. Зазвичай зберігають 7-30 днів.
Як забезпечити GDPR right-to-be-forgotten у Data Lake?
В "голому" Parquet-lake це неможливо без повної переливки. З Iceberg: 1) DELETE FROM table WHERE user_id = X — row-level delete, Iceberg записує delete file (positional або equality delete), не одразу перезаписує Parquet. 2) CALL expire_snapshots() — видаляє старі snapshot-и де ще були дані. 3) CALL rewrite_data_files(where=...) — фізично перезаписує Parquet файли без рядків цього user_id. Тільки після всіх трьох кроків дані фізично видалені. Важливо також видалити з raw/bronze шару, з Kafka retention і з backup-ів. Документувати все у GDPR deletion log.
Що таке schema evolution у lakehouse-форматах?
Schema evolution = можливість змінити схему таблиці без міграції і без поломки читачів. Iceberg підтримує: ADD COLUMN (нові рядки матимуть колонку, старі — null або default), DROP COLUMN (читачі отримають null), RENAME COLUMN (через ID-based mapping, а не по імені), REORDER COLUMNS, UPDATE TYPE (int → long, float → double — safe widening only). Ключова фішка Iceberg: колонки ідентифікуються за ID, а не ім'ям — перейменування не ламає планів запитів. Delta Lake аналогічно через schema merging. Порівняно з Hive: зміна схеми в Hive ламала читачів або потребувала ALTER TABLE + MSCK REPAIR.

Data Warehouses

Data Warehouse — колонкова СУБД, оптимізована для аналітичних запитів (OLAP): де OLTP-база зберігає рядки і оновлює їх транзакційно, DWH зберігає стовпці і сканує мільярди рядків за секунди. Snowflake, BigQuery, Redshift і ClickHouse — чотири архітектурно різні підходи до цієї задачі. Поверх DWH — dbt: інструмент, що перетворює SQL-файли на версіоновані, тестовані, документовані трансформації даних.

Коротка історія DWH

  • 1976-1990-і — Teradata: перша колонкова MPP-система на спеціалізованому залізі. Банки, телеком.
  • 1990-і — Bill Inmon і Ralph Kimball публікують книги про DWH-архітектуру і dimensional modeling. Ми досі живемо у їхній парадигмі.
  • 2000-і — Oracle Exadata, IBM Netezza, Vertica, Greenplum. MPP стає mainstream.
  • 2012 — Amazon Redshift: перший cloud-native DWH. На порядок дешевше за legacy. Але compute і storage ще зчеплені.
  • 2014-15 — Snowflake: розділили compute і storage. Можна паралельно запускати 10 virtual warehouses на одному datase без конкуренції за ресурси. Революція.
  • 2012-2019 — BigQuery: повністю serverless. Платиш за обсяг оброблених байт, не за сервер.
  • 2016+ — ClickHouse: open-source колонкова БД від Яндексу. Божевільна швидкість, self-hosted. Популярно в СНД/Європі.
  • 2022+ — Databricks SQL, Firebolt, DuckDB як embedded-DWH.

Snowflake — архітектура, яка змінила ринок

Ключова відмінність Snowflake — повне розділення трьох шарів: storage (власні micro-partitions у S3/Azure Blob/GCS), compute (віртуальні warehouses — кластери T-shirt size: XS/S/M/L/XL...), services (metadata, security, query optimizer). Це дозволяє:

  • Запустити 10 окремих warehouses на тих самих даних без конкуренції за ресурси (різні команди не ламають одна одній SLA).
  • Masштабувати compute вертикально (розмір) і горизонтально (кількість кластерів) незалежно.
  • Платити тільки за час реального використання — warehouse можна auto-suspend через 1 хв бездіяльності.
  • Zero-copy cloning — створити копію продового DWH за 2 секунди без дублювання даних (тільки metadata copy).
  • Time travel і fail-safe (7 днів додатково) — retention для recovery.

Особливість ціноутворення: тарифікація у credits. XS warehouse = 1 credit/год, S = 2, M = 4, L = 8, XL = 16. На AWS 1 credit ≈ $2-3 (standard) або $4 (enterprise). Запит на L-warehouse 10 хв = 1.3 credit ≈ $3. Помилка багатьох — залишити warehouse без auto-suspend.

BigQuery — serverless до кінця

  • Немає поняття «інстанс». Пишете SQL — Google Dremel виконує, ви платите за оброблені байти.
  • Ціна (on-demand): $6.25 за TB сканованих (us-central1). Тому завжди використовуйте partitioning + clustering, щоб обмежити скан.
  • Альтернатива — Capacity pricing (раніше Reservation): купуєте slots (compute units) на годину/місяць/рік. Має сенс при > $2000/міс on-demand.
  • Partitioning — тільки за 1 колонкою (типово date). Clustering — до 4 колонок для micro-pruning в межах партиції.
  • Нативна інтеграція з GCS (external tables), Google Sheets, Looker.
  • BigLake — читання Iceberg/Delta з GCS як external tables.

Redshift — для AWS-стеку

  • Historical — leader-based MPP, треба було вручну розбиратись з distribution keys і sort keys.
  • Сучасний — RA3 nodes з managed storage, плюс Redshift Serverless (2022).
  • Сильна інтеграція з AWS: Spectrum для читання S3 Parquet як external tables, Zero-ETL з Aurora.
  • Ціна: $0.25-$13.04/год залежно від розміру вузла; serverless — $0.375/RPU-год.
  • Зазвичай програє Snowflake і BigQuery по developer experience, але виграє, коли ви вже у глибині AWS і потрібен tight integration.

ClickHouse — self-hosted швидкість

  • Open-source колонкова OLAP БД. Створена Яндексом для аналітики метрики; зараз — ClickHouse Inc.
  • Сильні сторони: мільйони рядків/сек, real-time inserts, MergeTree-family движків.
  • Підтримує Kafka engine, S3 engine, materialized views зі стрімовими оновленнями.
  • Мінуси: немає повноцінних ACID-транзакцій, UPDATE/DELETE — повільні (mutations), JOINs слабші за Snowflake/BigQuery.
  • Сценарії: product analytics (PostHog on ClickHouse), observability (SigNoz), time-series, user-facing analytics дашборди.
  • Є managed-версія ClickHouse Cloud (2022). Ціна — $0.22-0.44/compute-unit-hour + $0.047/GB-місяць storage.

🎯 Interview Q&A — Модуль 7: DWH + dbt
Що таке dbt і чому він став стандартом у modern data stack?
dbt (data build tool) — фреймворк для трансформацій у DWH. Ідея: аналітики пишуть тільки SELECT-запити (models), dbt сам вирішує порядок виконання через граф залежностей (DAG через ref()), матеріалізує результати (table, view, incremental), запускає тести (unique, not_null, relationships), генерує документацію. До dbt трансформації були: хаотичні SQL-скрипти в Airflow, stored procedures без тестів, views без documentation. dbt зробив DWH-шар version-controlled і testable. Переваги: 1) SQL (не Python) — аналітики без engineering-досвіду можуть контрибьютити, 2) Incremental materializations для production-scale, 3) Tests для data quality, 4) Auto-generated lineage graph.
Що таке dbt incremental і як він обробляє дублікати?
Incremental materialization: перший run — будує таблицю повністю. Наступні — обробляє тільки нові рядки (WHERE event_date > max_in_target). Це ключово для великих таблиць де повна перебудова коштує $100/run. Дублікати вирішуються через unique_key: якщо вказаний, dbt робить MERGE (upsert): нові рядки — INSERT, існуючі — UPDATE. Без unique_key — тільки INSERT нових. Важливі конфіги: on_schema_change='append_new_columns' (додати нову колонку при schema change), merge_update_columns=['col1','col2'] (оновлювати тільки ці колонки). Пітфол: якщо late-arriving data (events з минулого) — потрібен lookback window у WHERE clause.
Snowflake vs BigQuery — як обирати?
Snowflake: compute/storage відокремлені, multiple warehouses на тих самих даних (multi-cluster), credits/год. Краще для: enterprise, мультихмара, складний SQL, команди з різними навантаженнями. BigQuery: повністю serverless (немає "warehouses"), платиш за TB сканованих. Краще для: ad-hoc аналітика, вже у GCP, ML pipeline через BigQuery ML. Практично: якщо у вас є постійне навантаження — Snowflake predictable cost. Якщо sporadic запити — BigQuery виграє (платиш тільки за запити). Обидва читають Iceberg з S3/GCS через external tables. Для стартапу без precommitment — BigQuery через on-demand pricing і free 10GB/month.
SCD Type 2 — поясніть механізм і типовий SQL
SCD2 зберігає повну історію змін атрибутів dimension. Кожна зміна = новий рядок з valid_from/valid_to/is_current. Структура: customer_key (surrogate, auto-increment), customer_id (natural key), email, country, valid_from TIMESTAMP, valid_to TIMESTAMP (NULL = поточний), is_current BOOLEAN. При зміні: закрити старий рядок (SET valid_to = NOW(), is_current = FALSE) + додати новий (valid_from = NOW(), valid_to = NULL, is_current = TRUE). Fact join: ON f.customer_key = d.customer_key — surrogate key не змінюється. Для правильного join по часу: ON f.customer_id = d.customer_id AND f.order_date BETWEEN d.valid_from AND COALESCE(d.valid_to, '9999-12-31'). dbt реалізує через dbt_utils.generate_surrogate_key() + snapshot mechanism.
Як тестувати якість даних у DWH?
dbt тести: 1) Generic tests — unique, not_null, accepted_values, relationships. Декларативно в YAML, запускаються після кожного run. 2) Singular tests — кастомний SQL який повертає рядки-порушення (SELECT order_id FROM fct_orders WHERE total_amount_usd < 0). 3) dbt-expectations (пакет) — expect_column_values_to_be_between, expect_table_row_count_to_be_between. 4) Source freshness — dbt source freshness перевіряє чи нові дані є у source. Окрім dbt: Great Expectations для глибшого profiling, Monte Carlo для ML-based anomaly detection, Soda для scheduled checks у production. Стратегія: dbt tests блокують deployment (CI/CD), production alerts через Soda або Monte Carlo.
Що таке партиціонування і кластеризація у BigQuery?
Partitioning: розподіл таблиці на незалежні сегменти за partition column (зазвичай DATE). BigQuery зберігає кожну партицію окремо — запит з WHERE date = '2026-01-01' сканує тільки 1 партицію замість усієї таблиці. Економія: 90%+ скану = 90% коштів. Clustering: сортування даних в межах партиції за 1-4 колонками (country, product_id). При запиті з WHERE country = 'UA' BigQuery знає, де фізично ці рядки — ще менше скану. Разом: PARTITION BY DATE(order_date) CLUSTER BY country, status. Best practice: майже завжди partition by date + cluster by найчастіший WHERE/GROUP BY. Якщо JOIN по user_id — cluster by user_id для colocation.
Що таке data lineage і навіщо він потрібен?
Data lineage — граф, який показує звідки прийшли дані і що з ними відбулось на кожному кроці. Від raw source → через трансформації → до BI dashboard. Навіщо: 1) Impact analysis — якщо змінити upstream таблицю, які downstream дашборди зламаються? 2) Root cause analysis — звіт показує неправильні цифри → простежити назад до source, 3) Compliance/audit — звідки взялися ці дані для регулятора, 4) Onboarding — нові люди розуміють пайплайн. Інструменти: dbt docs generate (статичний lineage граф), DataHub (enterprise-grade, OpenLineage API), OpenMetadata (open-source), Marquez. dbt артефакти (manifest.json) можна завантажити у DataHub через datahub ingest -c dbt.yml.

Orchestration & NoSQL

Apache Airflow — платформа оркестрації, де кожен pipeline описується як Python-клас (DAG) зі списком задач і залежностями між ними. MongoDB, Cassandra, Redis, Elasticsearch — чотири класи NoSQL-сховищ, кожен оптимізований під свій паттерн доступу: документи, wide-column, key-value in-memory, повнотекстовий пошук. Разом вони закривають задачі, де реляційна БД або занадто повільна, або не масштабується горизонтально.

Навіщо взагалі orchestrator

У вас є 15 Spark-джобів, 3 dbt-models, міграція SCD2, compaction Iceberg і експорт у BI. Вони повинні виконуватись у певному порядку (спочатку bronze→silver, потім silver→gold, потім DWH), з обробкою помилок (retry 3 рази з exponential backoff), з моніторингом SLA («fct_orders має бути готовим до 07:00 щодня»), з можливістю backfill за минулий тиждень після багфіксу. Це не «написати cron» — це повноцінна система. Саме цим і займаються orchestrator-и.

Apache Airflow — де-факто стандарт

Airflow створили в Airbnb у 2014, у 2019 став Apache top-level. До 2026 — найпоширеніший orchestrator (MWAA на AWS, Cloud Composer у GCP, Astro і Astronomer у ентерпрайзі).

Ключові концепції
  • DAG (Directed Acyclic Graph) — ваш pipeline. Python-файл, що описує таски та їхні залежності.
  • Task — одна одиниця роботи. Екземпляр Operator-а.
  • Operator — «шаблон» задачі: PythonOperator, BashOperator, SparkSubmitOperator, KubernetesPodOperator, BigQueryOperator. Свої custom-оператори — норма.
  • Sensor — спеціальний тип task-а, який чекає події (файл у S3, партиція у BigQuery, row у базі). Використовуйте з mode='reschedule'.
  • Scheduler — сервіс, що триггерить DAG-и за розкладом (cron-expression або @daily).
  • Executor — виконує таски. LocalExecutor (один сервер), CeleryExecutor (паралельно на багатьох worker-ах), KubernetesExecutor (кожен task — свій pod).
  • XCom — мікро-стор для обміну даними між тасками (не передавайте через нього великі об'єми — тільки метадані).

CAP-теорема · реальний вибір між CP і AP

CAP-теорема теоретична: у реальних системах усе складніше (PACELC розширення: навіть без partition обираєте між Latency і Consistency). Але для розуміння NoSQL-світу базу треба знати.

NoSQL — чотири великі категорії

D
Document

MongoDB · Couchbase · AWS DocumentDB. JSON/BSON документи з гнучкою схемою. Для user profiles, content management, каталогу продуктів.

W
Wide-column

Cassandra · ScyllaDB · HBase · BigTable. Масивний scale, оптимізований write path. Для time-series, логів, IoT, recommendation engines (Netflix — Cassandra).

K
Key-Value

Redis · DynamoDB · KeyDB · Valkey. Найпростіша модель, максимальна швидкість. Для кешів, session storage, leaderboards, real-time лічильників.

G
Graph

Neo4j · ArangoDB · Amazon Neptune. Відносини як first-class. Для recommendation, social networks, fraud detection, knowledge graphs.

Time-series БД — окрема історія

  • InfluxDB — OSS, спеціалізована. Monitoring, IoT. Має власну SQL-подібну мову Flux.
  • TimescaleDB — розширення PostgreSQL. Якщо вже є Postgres і часові ряди — найкращий вибір. SQL знайомий усім.
  • QuestDB — новіша, високопродуктивна для financial data.
  • Prometheus — не загального призначення, а для метрик інфраструктури (про це у Частині 3).
  • Альтернатива: ClickHouse з time-based partitioning — часто швидше за спеціалізовані time-series БД.

Коли SQL, коли NoSQL

SQL краще коли...
  • Сильна реляційна модель, складні joins
  • ACID-транзакції важливі
  • Схема стабільна, мало змінюється
  • Команда знайома з SQL (завжди)
  • Потрібні звіти і ad-hoc-аналітика
NoSQL краще коли...
  • Гнучка схема / швидко еволюціонує
  • Горизонтальне масштабування понад усе
  • Специфічний pattern доступу (графи, key-value, time-series)
  • Дуже high write throughput
  • Global distribution з eventual consistency ОК

from airflow.decorators import task, dag
from airflow.sensors.s3_key_sensor import S3KeySensor

    SlackWebhookOperator(
        task_id="slack_alert",
        slack_webhook_conn_id="slack_shopwave",

    catchup=False,

    wait_for_bronze = S3KeySensor(

        bucket_name="shopwave-prod-lake",
        bucket_key="bronze/orders/year={{ ds[:4] }}/month={{ ds[5:7] }}/day={{ ds[8:10] }}/_SUCCESS",
        aws_conn_id="aws_default",
        timeout=3600,              # чекаємо max 1 годину
        poke_interval=60,
        mode="reschedule",         # не займати worker slot під час очікування
        sla=timedelta(hours=2),
    )

    # 2. Bronze → Silver через Spark
    bronze_to_silver = SparkSubmitOperator(
        task_id="bronze_to_silver",
        application="s3://shopwave-code/spark/bronze_to_silver.py",
        conn_id="spark_k8s",
        application_args=["--date", "{{ ds }}"],
        conf={
            "spark.executor.memory": "4g",
            "spark.executor.instances": "8",
            "spark.sql.extensions": "org.apache.iceberg.spark.extensions.IcebergSparkSessionExtensions",
        },
        sla=timedelta(hours=1),
    )

    # 3. Validation — чи є достатньо рядків у silver?
    @task
    def validate_silver(ds: str) -> str:
        from pyspark.sql import SparkSession
        spark = SparkSession.builder.getOrCreate()
        count = spark.sql(
            f"SELECT COUNT(*) FROM shopwave.silver.orders WHERE order_date = '{ds}'"
        ).collect()[0][0]
        # Branching: якщо менше 1000 рядків → підозріло
        return "load_to_dwh" if count >= 1000 else "alert_low_volume"

    validation = validate_silver()

    # 4. Alert якщо мало даних
    alert_low_volume = SlackWebhookOperator(
        task_id="alert_low_volume",
        slack_webhook_conn_id="slack_shopwave",
        message=":warning: Silver layer має < 1000 рядків за {{ ds }}. Перевірте pipeline.",
    )

    # 5. Load Silver → DWH (ClickHouse)
    @task(sla=timedelta(minutes=30))
    def load_to_dwh(ds: str) -> dict:
        from clickhouse_driver import Client
        client = Client("clickhouse.shopwave.svc")
        result = client.execute(
            "INSERT INTO shopwave.fct_orders SELECT * FROM iceberg('s3://...', 'silver/orders') "
            "WHERE order_date = %(date)s",
            {"date": ds}
        )
        return {"rows_loaded": result}

    # 6. dbt run після завантаження
    @task
    def run_dbt(ds: str, load_result: dict) -> None:
        import subprocess
        proc = subprocess.run(
            ["dbt", "run", "--select", "marts", "--vars", f'{{"ds": "{ds}"}}'],
            capture_output=True, text=True, cwd="/opt/dbt/shopwave"
        )
        if proc.returncode != 0:
            raise RuntimeError(f"dbt failed:\n{proc.stderr}")

        task_id="notify_success",
        slack_webhook_conn_id="slack_shopwave",
        message=":white_check_mark: ShopWave ETL {{ ds }} completed successfully.",
        trigger_rule=TriggerRule.ALL_SUCCESS,
    )

    # Залежності
    silver_ready = bronze_to_silver
    branch = BranchPythonOperator(
        task_id="validate_silver",
        python_callable=validate_silver,
    )

    wait_for_bronze >> bronze_to_silver >> branch
    branch >> alert_low_volume
    branch >> load_to_dwh("{{ ds }}", {}) >> run_dbt("{{ ds }}", {}) >> notify_success

🎯 Interview Q&A — Модуль 8: Airflow + NoSQL
Що таке idempotency у DAG і чому це критично?
Idempotent task: виконання N разів дає той самий результат, що й одноразове виконання. Критично бо Airflow може retry task при падінні або перезапуску. Не-ідемпотентний task: INSERT INTO table SELECT ... — дублікати при retry. Ідемпотентний: INSERT INTO table SELECT ... WHERE date = '{{ ds }}' + unique constraint або MERGE/upsert. Практики: 1) Завжди використовувати {{ ds }} (execution date) а не поточну дату NOW(), 2) Замість INSERT — UPSERT (ON CONFLICT DO UPDATE), 3) Для Parquet writes — mode("overwrite") для конкретної партиції, 4) Для зовнішніх API — check-before-write: чи не відправляли email вже, 5) Зберігати стан у dedicated state table з execution_date.
XCom у Airflow — що це і коли використовувати?
XCom (Cross-Communication) — механізм передачі невеликих значень між tasks у межах DAG run. Task A пуш-ає XCom: ti.xcom_push(key='count', value=1234). Task B читає: ti.xcom_pull(task_ids='task_a', key='count'). З TaskFlow API (@task декоратор) XCom автоматичний: return value = XCom push, параметр функції = XCom pull. Коли використовувати: передати metadata між tasks (кількість рядків, URL файлу, статус). Коли НЕ використовувати: передавати DataFrame або великі дані — XCom зберігається в метадата-БД Airflow (обмежено до ~1MB). Для великих даних — пишіть в S3/GCS і передавайте URL. Пітфол: XCom coupling ускладнює тестування tasks окремо.
Dynamic DAGs в Airflow — що це і як реалізувати?
Dynamic DAG = програмно генерувати кілька DAG-ів або task-ів на основі конфігурації. Use cases: 1) Один DAG-шаблон для кожної таблиці (20 таблиць = 20 DAGs), 2) Паралельна обробка кількох країн як окремих tasks. Реалізація через .generate_tasks(): for country in ['UA','PL','DE']: task = SparkSubmitOperator(task_id=f'process_{country}', ...). З Airflow 2.5+ — Task Mapping через .expand(): process.expand(country=['UA','PL','DE']) — кожна країна як динамічний mapped task. Це ефективніше ніж generate_tasks бо Airflow знає кількість tasks тільки в runtime. Проблема з dynamic DAGs: DagBag parsing при кожному scheduler tick — тисячі DAGs сповільнюють scheduler. Рішення: кешувати конфіг.
MongoDB vs PostgreSQL — коли NoSQL виправданий?
PostgreSQL: ACID, SQL joins, fixed schema, FK constraints. Ідеально для: фінансові транзакції, складні аналітичні запити з JOIN-ами, нормалізовані дані. MongoDB: BSON documents (nested JSON), flexible schema, horizontal sharding, денормалізовані дані. Ідеально для: 1) user profiles з різноманітною структурою (різні поля для різних типів юзерів), 2) product catalogs де кожна категорія має різні attributes (телефон vs одяг), 3) session/activity store де структура нестабільна, 4) швидкі writes за document key. Антипатерн MongoDB: транзакції між кількома documents (погана підтримка), складні аналітичні агрегації (краще OLAP), master-data з FK. Загальне правило: якщо сумніваєтесь — PostgreSQL. MongoDB тільки коли flexible schema — справжня вимога.
Як правильно налаштувати Airflow для production?
Ключові налаштування: 1) Executor: LocalExecutor (1 машина, <20 tasks паралельно), CeleryExecutor (worker pool, масштабується), KubernetesExecutor (кожен task = K8s Pod, ізоляція). 2) Connections: ніколи не хардкодити credentials — Airflow Connections у secrets backend (AWS SSM, Vault, K8s Secrets). 3) SLA: sla=timedelta(hours=2) на critical tasks — Airflow надішле email якщо task не завершився. 4) Pool: обмежити паралельні connections до DWH: Pool('dwh_queries', slots=10). 5) DAG versioning: при зміні DAG структури — новий dag_id, не edit старого (зламає backfill). 6) Monitoring: Statsd metrics → Prometheus → Grafana. Ключові метрики: scheduler_heartbeat, dag_processing_total, task_duration.
CAP теорема — як застосовувати при виборі БД?
CAP: Consistency (всі вузли бачать одні дані), Availability (кожен запит отримує відповідь), Partition tolerance (система працює при network split). В розподіленій системі P завжди потрібен — тому обираємо CP або AP. CP (Consistency + Partition tolerance): MongoDB (з write concern majority), HBase, Redis Cluster. При network partition — система відхиляє writes і стає недоступною, але дані консистентні. AP (Availability + Partition tolerance): Cassandra, DynamoDB, CouchDB. При partition — відповідає з potentially stale даними, але ніколи не падає. Практично: банківський рахунок → CP (краще помилка ніж неправильний баланс).
